## Summary

Phase 2.5 is a small, mostly-mechanical rebrand from the working name `voice-cc` to `PurpleVoice` (PascalCase brand) / `purplevoice` (lowercase module/binary/path). Every load-bearing decision is locked in CONTEXT.md — research's job is to fill the technical implementation gaps so the planner can write atomic plan tasks the executor can verify.

The seven research gaps in the prompt resolve cleanly: (1) `hs.styledtext.new("●", { color = { hex = "#B388EB" } })` is already the established pattern in the existing code — the rebrand is a hex-string swap, with the recording-state styling difference handled via a second styledtext object using a complementary lavender shade or font weight. (2) `sips` on macOS Sequoia 15.7.5 converts SVG → 256×256 PNG natively (verified empirically during research) — no external dependencies needed; the icon ships as a small hand-authored SVG checked into `assets/` and a deterministic sips conversion produces the PNG. (3) The idempotent migration block uses `mv` on directories with explicit four-state guards (only-old, both, only-new, neither). (4) Per-file targeted edits with grep verification beats a global sed pass — the Pattern 2 boundary (`grep -c WHISPER_BIN voice-cc-record == 2`) is the canonical example of an invariant that benefits from targeted review, and the rebrand introduces a similar invariant (`grep -c "voice-cc" repo == 0` after, except in approved-list files). (5) Hammerspoon notification callback re-registration is the documented mechanism for surviving reload — old tags become orphaned but cause only "console raise on click", not crashes; the rebrand should call `hs.notify.unregister()` on the old tags as a one-line cleanup. (6) PROJECT.md update is surgical: replace 4 places with `PurpleVoice`, append the brand-rationale paragraph + privacy-positioning paragraph. (7) BRD-01..03 acceptance criteria are best expressed as grep predicates and a file-existence check.

**Primary recommendation:** Plan Phase 2.5 as four atomic plans aligned with STATE.md's predicted shape — (a) string propagation in code surfaces, (b) XDG path rename + idempotent migration, (c) visual identity (menubar lavender + 256×256 PNG via sips from SVG), (d) README + tagline + REQUIREMENTS.md BRD-01..03 elaboration + PROJECT.md update. **Recommended icon-generation path:**

1. Hand-author a minimal SVG (~15-25 lines, viewBox `0 0 256 256`, one rect for background `#B388EB`, one path for lips silhouette, white fill). Check `assets/icon.svg` into repo.
2. Run `/usr/bin/sips -s format png --resampleHeightWidth 256 256 assets/icon.svg --out assets/icon-256.png` once. Check the output PNG into repo.
3. Document the regenerate command in a `assets/README.md` comment (single line) so the icon is reproducible if the SVG is edited later. No Makefile needed for a personal tool.

This means the SVG is the source of truth and the PNG is a derived artifact, but both are committed for ergonomics — `sips` is a one-time invocation, not an automated build pipeline.

### Pattern 1: Per-File Targeted Rebrand with Grep Verification

**What:** For each file in scope, do a deliberate per-line edit (or whole-file rewrite where it's a pure string substitution), then run `grep -c "voice-cc" <file>` and verify the count is 0 (or matches an approved exception count for files where partial preservation is intentional).

**When to use:** Any file with `voice-cc` references that needs a rebrand. The grep-count assertion is the validation primitive.

**Why preferred over global sed:**
1. The Pattern 2 boundary (`grep -c WHISPER_BIN voice-cc-record == 2`) is a separate invariant that must survive — deliberate per-file editing keeps the bash glue's structural integrity visible.
2. Some `voice-cc` references in test scripts reference the *old* notification titles in error messages or test fixtures — those need contextual judgement (e.g., a comment that says "test_tcc_notification.md asserts the title is 'voice-cc: microphone blocked'" must update both the assertion AND the title comment together; sed would just fix the latter and leave the former lying about the actual code behaviour).
3. setup.sh's migration block needs both old and new path strings to coexist (it's literally migrating `voice-cc` paths to `purplevoice` paths) — sed would rewrite both sides.

**Example flow per file:**
```bash
# Before edit
grep -c "voice-cc" "$FILE"                    # baseline count

# Edit the file (Edit tool or careful sed)

grep -n "voice-cc" "$FILE"                    # surviving refs (should be 0 unless approved)
grep -c "purplevoice" "$FILE"                 # should be ≥ baseline old count
```

**Approved exception list (files where `voice-cc` is intentionally preserved):**
- All files under `.planning/` (per D-07)
- `setup.sh` migration block (specifically the lines that reference the old paths to migrate FROM)
- Any line in source comments or documentation that historically describes the old name (e.g., a CHANGELOG entry, if one exists — none does currently)

### Pattern 2: Idempotent setup.sh Migration Block

**What:** Single bash function in setup.sh that handles all four states (only-old, both, only-new, neither) for each XDG directory.

**When to use:** D-05's one-time migration. Runs on every setup.sh invocation but is a no-op if the user has already migrated.

**Example (canonical form):**
```bash
# In setup.sh, BEFORE Step 4 (mkdir -p new XDG paths)
migrate_xdg_dir() {
  local old="$1"
  local new="$2"
  local label="$3"

  if [ ! -d "$old" ] && [ ! -d "$new" ]; then
    return 0  # neither exists — fresh install, nothing to migrate
  fi

  if [ ! -d "$old" ] && [ -d "$new" ]; then
    return 0  # only-new — already migrated or fresh install on new path
  fi

  if [ -d "$old" ] && [ -d "$new" ]; then
    echo "WARN: both $old AND $new exist — leaving both. Resolve manually." >&2
    return 0  # do NOT clobber either side
  fi

  # only-old → migrate
  echo "Migrating $label: $old → $new"
  mkdir -p "$(dirname "$new")"
  mv "$old" "$new"
}

migrate_xdg_dir "$HOME/.config/voice-cc"        "$HOME/.config/purplevoice"        "config"
migrate_xdg_dir "$HOME/.local/share/voice-cc"   "$HOME/.local/share/purplevoice"   "data (~190 MB models)"
migrate_xdg_dir "$HOME/.cache/voice-cc"         "$HOME/.cache/purplevoice"         "cache"

# Also relink the bash binary symlink
if [ -L "$HOME/.local/bin/voice-cc-record" ]; then
  rm "$HOME/.local/bin/voice-cc-record"
fi

# Also relink the Hammerspoon module symlink
if [ -L "$HOME/.hammerspoon/voice-cc" ]; then
  rm "$HOME/.hammerspoon/voice-cc"
fi
```

**Why `mv` not `rsync`:**
- All XDG dirs are under `$HOME` on a single APFS volume → `mv` is atomic at the directory inode level, no copy phase.
- D-05 explicitly says "move not copy" to avoid the 190 MB model duplication.
- `rsync -a --remove-source-files` would do per-file copies + per-file delete, which on a 488 MB model file is ~10 seconds of unnecessary IO.
- Edge cases: if the user has put symlinks INTO `~/.config/voice-cc/` themselves, `mv` preserves them (it's a directory-rename, not a tree-walk).
- Partial-migration recovery: if `mv` is interrupted by SIGKILL mid-rename of a directory (vanishingly unlikely on a single-volume rename), the user is left in an inconsistent state. For a one-user one-shot migration on a personal machine, this risk is acceptable. Document the recovery: "if interrupted, manually inspect `~/.config/voice-cc/` and `~/.config/purplevoice/`, move whichever has the data into the new path, delete the empty one."

### Pattern 3: hs.styledtext Lavender Glyph (recording-state difference)

**What:** Two `hs.styledtext` objects, both using `#B388EB` lavender, with a structural styling difference for recording state.

**When to use:** Replacing the existing grey/red dot in `voice-cc-lua/init.lua` lines 39-40, 56-57.

**Recommended pattern (preserves existing code structure):**
```lua
-- Constants block (replace MENUBAR_IDLE_COLOR / MENUBAR_RECORDING_COLOR)
local MENUBAR_LAVENDER = "#B388EB"

-- Two styledtext objects — same colour, different weight (idle vs bold for recording).
-- The bold weight is a Hammerspoon-native trait change via convertFont.
local idleTitle = hs.styledtext.new("●", { color = { hex = MENUBAR_LAVENDER } })
local recordingTitle = hs.styledtext.new("●", {
  color = { hex = MENUBAR_LAVENDER },
  font = { name = "Menlo-Bold", size = 14 },  -- bolder weight for recording
})
```

**Alternatives within the lavender palette (Claude's discretion):**
- **Filled vs outline glyph:** idle = `○` (white circle, U+25CB), recording = `●` (black circle, U+25CF) — both styled lavender. Visually clearer than weight difference.
- **Two shades of purple:** idle = `#B388EB` (lavender), recording = `#7E54C9` (deeper purple). Stays in the brand palette but differentiates more strongly.
- **Same glyph + opacity:** idle = `#B388EB` at alpha 0.5, recording = `#B388EB` at alpha 1.0 — subtle but present. Use `{ hex = "#B388EB", alpha = 0.5 }` per the verified hex format.

Recommended: **filled vs outline glyph** approach. Visually clear, no font dependency, single colour constant.

### Anti-Patterns to Avoid

- **Global `find . -exec sed` rebrand pass.** Will silently rewrite `.planning/` content (violates D-07) and will rewrite both sides of the setup.sh migration block (breaking the migration). Always per-file targeted edits.
- **Renaming the repo working directory.** D-07 explicitly preserves this. The path `/Users/oliverallen/Temp video/voice-cc/` stays.
- **Editing the user's `~/.hammerspoon/init.lua` automatically.** ARCHITECTURE.md Anti-Pattern 4 — never auto-edit user dotfiles. Setup.sh prints the new line; user pastes.
- **Adding ImageMagick / Inkscape / Python as a build dependency for the icon.** `sips` is built-in. No new dependencies.
- **Forgetting to unregister old hs.notify tags.** The `voiceccOpenMicSettings` and `voiceccOpenAccessibilitySettings` callbacks should be unregistered during the rebrand to avoid orphaned-tag console-raise behaviour on any old notification still in the macOS Notification Center. One-line cleanup at the top of the new init.lua: `pcall(hs.notify.unregister, "voiceccOpenMicSettings"); pcall(hs.notify.unregister, "voiceccOpenAccessibilitySettings")`.
- **Mixing brand casings.** `PurpleVoice` for user-facing prose, `purplevoice` for module/binary/path. Never `purpleVoice`, `Purplevoice`, `PURPLE_VOICE`, etc. (Existing env var pattern is `VOICE_CC_*`; the rebrand should change to `PURPLEVOICE_*` or `PV_*` — recommend `PURPLEVOICE_*` for consistency with the lowercase brand spelling.)

## Common Pitfalls

### Pitfall 1: Sed-based rebrand wipes setup.sh migration block

**What goes wrong:** A `find . -exec sed -i 's/voice-cc/purplevoice/g'` pass would rewrite both sides of the migration block, so `mv "$HOME/.config/voice-cc" "$HOME/.config/purplevoice"` becomes `mv "$HOME/.config/purplevoice" "$HOME/.config/purplevoice"` (no-op, then the migration silently does nothing).

**Why it happens:** sed has no semantic awareness of "this file is the migration block, leave the FROM path".

**How to avoid:** Per-file Edit tool calls. For setup.sh specifically, write the migration block AFTER the rename pass and use literal `voice-cc` strings in the FROM positions deliberately.

**Warning signs:** After-rebrand grep `grep -n "voice-cc" setup.sh` returns 0 lines. That should be ≥3 lines (the three migrate_xdg_dir calls' FROM paths). If it's 0, the migration block was wiped.

### Pitfall 2: Symlink not recreated after rename

**What goes wrong:** Repo's `voice-cc-lua/` is renamed to `purplevoice-lua/`, but `~/.hammerspoon/voice-cc` still points at the old path → broken symlink → `require("voice-cc")` fails → Hammerspoon errors. Same for `~/.local/bin/voice-cc-record`.

**Why it happens:** `mv` of a symlink target doesn't update the symlink — it just orphans it.

**How to avoid:** Setup.sh's migration block must `rm` the old symlinks and recreate the new ones with the new names. Verify with `readlink ~/.hammerspoon/purplevoice` and `readlink ~/.local/bin/purplevoice-record` (both should resolve, not error).

**Warning signs:** Hammerspoon console shows `module 'voice-cc' not found`. Or `~/.local/bin/voice-cc-record: No such file or directory` when invoked.

### Pitfall 3: Pattern 2 boundary lost during edits

**What goes wrong:** During the rebrand of voice-cc-record, an editor "tidies" the WHISPER_BIN line (it's currently a one-liner with folded `[ -x ]` check) into multiple lines, breaking the `grep -c WHISPER_BIN == 2` invariant.

**Why it happens:** The Pattern 2 boundary is intentional but unusual-looking; an editor unfamiliar with the constraint might "improve" it.

**How to avoid:** Add `grep -c WHISPER_BIN purplevoice-record == 2` as an explicit verification step in any plan that touches this file. The same applies to `! grep -q whisper-cli purplevoice-lua/init.lua`.

**Warning signs:** Rebrand verification fails the existing Pattern 2 invariant.

### Pitfall 4: hs.notify tag rename leaves orphaned callbacks

**What goes wrong:** User had a TCC-deny test that left a notification in macOS Notification Center referencing tag `voiceccOpenMicSettings`. After rebrand, the new init.lua only registers `purplevoiceOpenMicSettings`. User clicks the OLD notification → Hammerspoon console raises (default behaviour for orphaned tags) instead of opening Settings.

**Why it happens:** Notification Center persists notifications across app restarts; tag registry is in-process state that resets on reload.

**How to avoid:** New init.lua calls `pcall(hs.notify.unregister, "voiceccOpenMicSettings")` and `pcall(hs.notify.unregister, "voiceccOpenAccessibilitySettings")` at the top. Both are no-ops if the tags don't exist. Optionally, also call `hs.notify.withdrawAll()` once (with user awareness — this nukes ALL Hammerspoon notifications, possibly undesirable).

**Warning signs:** Clicking a stale "voice-cc: microphone blocked" notification raises the console instead of opening Settings.

### Pitfall 5: User's manual `~/.hammerspoon/init.lua` edit forgotten

**What goes wrong:** User runs setup.sh, sees migration succeed, paths renamed, symlinks recreated. But their `~/.hammerspoon/init.lua` still has `require("voice-cc")` → Hammerspoon errors `module 'voice-cc' not found` because the symlink at `~/.hammerspoon/voice-cc` no longer exists.

**Why it happens:** ARCHITECTURE.md Anti-Pattern 4 — voice-cc never auto-edits user dotfiles. Setup.sh prints the new line; user must paste it.

**How to avoid:** Setup.sh's "next steps" output is loud and prescriptive: shows the OLD line, the NEW line, and the file path. Phase 3 will add automation for first-time installs but for this rebrand, manual edit is the established pattern.

**Warning signs:** Hammerspoon error after setup.sh completes successfully. User reports "everything broke".

### Pitfall 6: 256×256 PNG generated at wrong size

**What goes wrong:** sips `--resampleHeightWidth 256 256` works, but on a non-square SVG (e.g., viewBox `0 0 100 80`), sips will distort to fit the target. Or, the SVG's inherent dimensions get used and the PNG comes out at SVG-native size (e.g., 100×80) before resampling.

**Why it happens:** sips's SVG handling treats the SVG's `width` and `height` attributes as canonical; if the SVG omits them or uses `viewBox` only, dimension inference is unreliable.

**How to avoid:** Author the SVG with explicit `width="256" height="256" viewBox="0 0 256 256"` attributes. Verify after sips: `sips -g pixelWidth -g pixelHeight assets/icon-256.png` should report `256` for both.

**Warning signs:** Verify command reports any value other than 256.

### Idempotent migration block (setup.sh)

```bash
# Source: synthesised from D-05 + canonical bash idempotency patterns
migrate_xdg_dir() {
  local old="$1"
  local new="$2"
  local label="$3"
  if [ ! -d "$old" ] && [ ! -d "$new" ]; then return 0; fi
  if [ ! -d "$old" ] && [ -d "$new" ]; then return 0; fi
  if [ -d "$old" ] && [ -d "$new" ]; then
    echo "WARN: both $old AND $new exist — leaving both. Resolve manually." >&2
    return 0
  fi
  echo "Migrating $label: $old → $new"
  mkdir -p "$(dirname "$new")"
  mv "$old" "$new"
}

migrate_xdg_dir "$HOME/.config/voice-cc"      "$HOME/.config/purplevoice"      "config"
migrate_xdg_dir "$HOME/.local/share/voice-cc" "$HOME/.local/share/purplevoice" "data"
migrate_xdg_dir "$HOME/.cache/voice-cc"       "$HOME/.cache/purplevoice"       "cache"

# Symlink hygiene
[ -L "$HOME/.local/bin/voice-cc-record" ] && rm "$HOME/.local/bin/voice-cc-record"
[ -L "$HOME/.hammerspoon/voice-cc" ]      && rm "$HOME/.hammerspoon/voice-cc"
```
