## Implementation Decisions

### Product Name (BRD-01)

- **D-01:** **Product name = `PurpleVoice`** (single word, PascalCase as the canonical brand spelling). Lowercase `purplevoice` is the canonical form for module names, binary names, paths, and shell-friendly contexts.

  *Rationale:* Two-wave brainstorm explored English short/evocative names (Hush, Whisp, Pith, Magpie, Cricket, etc.) and anime/nerdy/vibe-cody names (Koe, Hayai, Bankai, etc.). Web availability checks killed most candidates because the dictation space is brutally crowded — Hush (Hush Touch), Pip (Pipit), Chirp (multiple), Koe (`koe.fm` — direct architectural twin) all collide. PurpleVoice is clean: no dictation/STT app uses the name; adjacent matches (a defunct VoIP company, a TTS voice, a voice-changer) are non-competing. The colour-noun compound is memorable, friendly, and gives the brand a clear visual hook (purple + voice-related glyph).

- **D-02:** **Positioning** — the target audience is **privacy-conscious users who do not want, or cannot use, cloud-connected dictation tools**. This is the load-bearing differentiator vs Koe (Doubao cloud default), Wispr Flow (subscription cloud), Voicy (cloud), and the broader market. Every user-visible string and the entire Phase 3 README narrative should reinforce *fully on-device, nothing leaves your Mac*.

### String Propagation (BRD-02)

- **D-03:** **Lua module name = `purplevoice`** (lowercase). User pastes `require("purplevoice")` into `~/.hammerspoon/init.lua`. Repo source dir renames `voice-cc-lua/` → `purplevoice-lua/`. Symlink from `~/.hammerspoon/purplevoice/` → `<repo>/purplevoice-lua/`.

- **D-04:** **Bash binary name = `purplevoice-record`** (lowercase, hyphen-separated). Repo source file renames `voice-cc-record` → `purplevoice-record`. Symlink from `~/.local/bin/purplevoice-record` → `<repo>/purplevoice-record`.

- **D-05:** **XDG paths rename + one-time migration in setup.sh.** Old paths `~/.config/voice-cc/`, `~/.local/share/voice-cc/`, `~/.cache/voice-cc/` migrate to `~/.config/purplevoice/`, `~/.local/share/purplevoice/`, `~/.cache/purplevoice/`. setup.sh's migration block is idempotent: if the old dir exists and the new doesn't, move; if both exist, leave both and warn; if only the new exists, no-op. Old `vocab.txt` and `denylist.txt` carry over verbatim. Old WAV models (~190 MB) move, not copy, to avoid duplication.

- **D-06:** **All Hammerspoon alerts and notification titles** are rewritten to use `PurpleVoice` (PascalCase, the canonical brand spelling) instead of `voice-cc`. Specifically:
  - Module load alert: `PurpleVoice loaded (cmd+shift+e)` (was `voice-cc loaded (cmd+shift+e)`)
  - All five notification titles: `PurpleVoice: microphone blocked` / `: install incomplete` / `: transcription failed` / `: accessibility required` / `: unexpected exit N`
  - Tagline appended where it fits (see D-13)

- **D-07:** **`voice-cc` is preserved verbatim in:** repo path on disk, git history, all `.planning/` artifacts (CONTEXT.md / SUMMARY.md / STATE.md / ROADMAP.md / phase directory names — e.g., `02-hardening` not renamed, `02.5-branding` directory is correct as-is). These are historical record. New planning artifacts written from this phase forward use `PurpleVoice` in user-facing prose but reference paths as they exist.

- **D-08:** **`.hammerspoon-init-snippet.lua`** (the staged snippet for the user to paste) updates to show the new `require("purplevoice")` line plus a comment line explaining the brand. Setup.sh's "next steps" output points at the updated snippet.

### Visual Identity (BRD-03)

- **D-09:** **Menubar dot colour = `#B388EB`** (lavender). Implemented via `hs.styledtext` with the colour applied to the existing `●` glyph. Idle and recording states use the same lavender hex; recording state additionally gets a styling difference (e.g., bolder weight or a switch to a filled-vs-outline glyph) — exact recording-state styling is **Claude's discretion** within the lavender palette.

- **D-10:** **App icon = 256×256 PNG, purple background with white lips.** Specifics:
  - Background: solid lavender `#B388EB` to match the menubar (or a slightly darker shade like `#7E54C9` for visual depth — Claude's discretion within the purple family)
  - Foreground: stylised white lips, centred, simple flat-design (no gradients), proportionate (~40-50% of canvas height)
  - File location: `assets/icon-256.png` in the repo (new `assets/` directory)
  - Used by: setup.sh banner display (printed via base64 or just referenced by path), README header image, and future macOS app-icon if voice-cc ever ships as a `.app`
  - **Generation tool: Claude's discretion** — could be hand-drawn in Figma export, generated via an AI image tool with the spec, or a simple SVG converted to PNG. The deliverable is the PNG file.

### Tagline & Placement (BRD-02 detail)

- **D-11:** **Canonical tagline = `"Local voice dictation. Nothing leaves your Mac."`** Two sentences, scan-friendly, leads with what (local dictation) and follows with the privacy guarantee.

- **D-12:** **Placement (all four):**
  - `README.md` header — appears as the line directly under the project name / H1
  - `setup.sh` banner — printed during install, after the icon
  - Hammerspoon load alert — extends to `PurpleVoice loaded — local dictation, cmd+shift+e` (slight rewording for the inline alert form factor)
  - Future Phase 3.5 HUD — reserved for the about-screen / tooltip; this phase only stages the string in a constants file the HUD can later import

### Claude's Discretion

- Icon generation tool / exact lip silhouette / exact background shade within purple family — spec is "purple background, white lips, 256×256 PNG"; visual choices stay within that.

## Existing Code Insights

### Reusable Assets

- `voice-cc-lua/init.lua` (315 lines, post-Phase-2 hardened) — All notification calls, alert text, and the menubar setup live here. The 5 notification titles use the literal prefix `voice-cc:` — a single sed-replace + manual review pass is the lowest-friction rebrand.
- `voice-cc-record` (176 lines) — Two `voice-cc:` strings in stderr error messages (whisper-cli missing, Whisper model missing). Path references for XDG dirs need updating in 8+ places.
- `setup.sh` — 16 occurrences of `voice-cc`. Banner block, path creation commands, model-download path, symlink targets. The migration block (D-05) is a new addition.
- `.hammerspoon-init-snippet.lua` — 3 occurrences. The snippet IS the user-visible "what to paste" example, so its rebrand is load-bearing.
- `README.md` — currently 2 occurrences (stub). Phase 2.5 expands README significantly — header, tagline, install instructions all touch the brand.
- `tests/test_*.sh` and `tests/manual/*.md` — scattered `voice-cc` references. The bash unit tests reference paths, manual walkthroughs reference notification titles. ALL need rebrand for consistency, even though the planning docs (`.planning/`) do not.

### Established Patterns

- **String references avoid hard-coding the brand in logic** — most `voice-cc` strings are in user-visible text or paths, not control flow. This makes rebranding mostly mechanical.
- **XDG layout (D-03 from Phase 1)** — the path family `~/.config|.local/share|.cache/<brand>/` is consistent throughout. Single point of variable change in setup.sh + bash glue.
- **Pattern 2 boundary preserved** — the WHISPER_BIN-on-two-lines invariant in `voice-cc-record` must stay intact through any rebrand-related edits. The variable name itself is fine (`WHISPER_BIN` is whisper-related, not brand-related).

### Integration Points

- Setup.sh is the single entry point for the rename + migration. New users get fresh paths; existing users (just Oliver) get one-shot migration.
- The user's `~/.hammerspoon/init.lua` is OUT of repo — they need to manually update `require("voice-cc")` → `require("purplevoice")` after the rename. setup.sh prints the new line at completion.
- macOS TCC permissions are bound to the *Hammerspoon process*, not to PurpleVoice's identifier — no TCC re-grant needed during rebrand. The Privacy & Security entry stays "Hammerspoon" forever (this is a Hammerspoon-architecture limitation, not a brand concern).

<specifics>
## Specific Ideas

- **The tagline reads cleanly out loud** — say "Local voice dictation. Nothing leaves your Mac." — that pacing works for the README header AND for narrating the project to someone in conversation.
- **PurpleVoice + lavender + lips imagery hangs together** — the colour, the icon, the brand name all point at the same metaphor (your voice, on your machine, intimately personal). Don't dilute by adding a microphone icon or other generic voice-app cliches.
- **Privacy framing is anti-Koe by design** — the entire reason the user chose to keep building rather than adopt Koe is *Koe defaults to cloud*. Lean into that. The README should explicitly compare: "If you've used [voice-cc-likes], you've trained their cloud. PurpleVoice is local, period."
- **Lavender (`#B388EB`)** specifically — not deep purple, not pastel violet. The hex is locked.
- **Icon spec: white lips, not a microphone** — chose lips because it's the human gesture (you speak with lips), where mic icons are generic and used by every dictation tool.

</specifics>
