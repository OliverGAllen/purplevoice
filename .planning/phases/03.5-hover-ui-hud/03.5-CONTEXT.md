# Phase 3.5: Hover UI / HUD — Context

**Gathered:** 2026-04-30
**Status:** Ready for planning

<domain>
## Phase Boundary

**What this phase delivers:** A small floating HUD that surfaces PurpleVoice recording state at-a-glance — a compact lavender pill with `● Recording` text appearing at the top-center of the active screen while the hotkey is held, hidden when idle. The HUD complements (does NOT replace) Phase 2's menubar indicator. Optimised for low CPU when idle, no focus stealing, no paste interference, and hidden from screen recordings by default.

**Phase 3.5 IS:**
- A new `hs.canvas` (or whatever API the researcher recommends) overlay rendered by `purplevoice-lua/init.lua`
- Hooked into the existing `onPress` (line 266) and `onRelease` (line 293) lifecycle alongside `setMenubarRecording` / `setMenubarIdle`
- Configurable via two new env vars (`PURPLEVOICE_HUD_OFF`, `PURPLEVOICE_HUD_POSITION`) read once at module load
- Hidden from screen recordings by default via `NSWindowSharingNone` (or Hammerspoon equivalent)
- Documented in README + SECURITY.md with an honest "best-effort" caveat about hide-from-recording reliability

**Phase 3.5 is NOT:**
- A replacement for the menubar indicator (which stays per Phase 2 D-09 — lavender outline ○ idle, filled ● recording)
- A settings UI (no GUI for configuration; env vars only)
- An interactive surface (no hover, no click, no right-click menu — it's an indicator, not a widget)
- A persistent visual (hidden when idle; only visible during press-hold)
- A brand-tweakable surface (colour locked to lavender per Phase 2.5 D-09; no colour-override env var)
- A new lifecycle layer (uses existing onPress/onRelease; no new hotkeys; no runtime toggle key)
- A tagline-integration surface for v1 (the canonical tagline `Local voice dictation. Nothing leaves your Mac.` already lives in README header / setup.sh banner / Hammerspoon load alert; HUD is too transient + non-interactive to host it usefully)

**Requirements covered (proposed in ROADMAP, formalised during planning):** HUD-01, HUD-02, HUD-03, HUD-04.

**Success criteria (from ROADMAP §"Phase 3.5"):**
1. While the hotkey is held, a floating HUD element appears with a visible "● Recording" affordance using the lavender brand from Phase 2.5.
2. The HUD disappears within ~250 ms of release; while idle, no HUD is on screen and no measurable CPU is consumed.
3. A config toggle hides the HUD entirely (`PURPLEVOICE_HUD_OFF=1`) for users who only want the menubar indicator (Phase 2 still ships the menubar regardless of HUD state).
4. The HUD does not steal focus, does not interfere with paste, and is hidden from screen recordings by default.

<decisions>
## Implementation Decisions

### Visual Form Factor

- **D-01:** **Form factor = pill.** Lavender translucent rounded-rectangle, ~140×36px, border-radius half-height (~18px) for cleanly-rounded pill. Centered text inside. This is the most common indicator-pill shape on macOS (System Settings, AirPods battery overlay, Now Playing) and gives the brand a distinctive silhouette against any background.

- **D-02:** **Content during recording = `● Recording`.** Filled circle glyph (U+25CF) followed by single space and the word `Recording`. Two-token comprehension. White text on lavender. Matches the menubar's filled-circle convention but adds a label since the HUD appears alone (no menubar context immediately adjacent).

- **D-03:** **Visual style = translucent lavender, NO backdrop blur.** Solid lavender `#B388EB` at alpha ~0.85. Modern semi-transparent look (some content visible behind) without the GPU cost or text-edge artifacts of a backdrop-blur (vibrancy) layer. User explicitly chose "option 3 without the blur" — this is the intent. Planner may tune the exact alpha within 0.80–0.90 if rendering needs adjustment, but no backdrop-blur layer.

- **D-04:** **Size = compact 140×36 px** at default macOS scaling. Big enough to read `● Recording` clearly at typical viewing distance; small enough to be unobtrusive. Sits comfortably at top-center or any named-position alternative.

### Default Position

- **D-05:** **Default position = top-center of active screen.** Horizontally centered, ~50px below the menubar. Out of the typing area, immediately visible in peripheral vision, doesn't compete with cursor or focused window.

- **D-06:** **Multi-monitor = active screen.** The screen containing the focused window (`hs.window.focusedWindow():screen()` at press time, with sensible fallback to primary if no focused window). Matches "where the user is working".

- **D-07:** **Position override via named positions only.** `PURPLEVOICE_HUD_POSITION=top-center | top-right | bottom-center | bottom-right | near-cursor | center`. Default `top-center`. Validates against the fixed list; on invalid value, falls back to `top-center` and prints a single warning to the Hammerspoon console at module load. No explicit-coordinate mode in v1 (avoids screen-dimension-arithmetic burden on the user; six named positions cover the realistic preference space).

- **D-08:** **Edge gap for `top-center` = ~50px below menubar.** macOS Sequoia menubar is ~25px tall; +25px gap = ~50px from screen top. Standalone visual, not bumped against the menubar; intentionally placed rather than appended.

### Configuration

- **D-09:** **Default state = ON.** First-run users see the HUD; no opt-in friction. `PURPLEVOICE_HUD_OFF=1` disables. Matches the existing `PURPLEVOICE_NO_SOUNDS` pattern (default-on, env var disables). Users who find it distracting flip one env var.

- **D-10:** **Two env vars, both with sensible defaults:**
  - `PURPLEVOICE_HUD_OFF=1` — hides the HUD entirely (menubar indicator still ships; Phase 2 unchanged)
  - `PURPLEVOICE_HUD_POSITION=<named>` — overrides default `top-center` (see D-07 for valid values)
  - No `PURPLEVOICE_HUD_ALPHA`, no `PURPLEVOICE_HUD_COLOR`, no offset overrides — exposing more knobs invites brand drift. Two knobs is the v1 surface.

- **D-11:** **No runtime toggle hotkey.** Env vars are read once at module load (matches `PURPLEVOICE_NO_SOUNDS` pattern in `purplevoice-lua/init.lua` line 99). Reload Hammerspoon to apply changes. Simpler, fewer code paths, no hotkey-collision risk with macOS shortcuts (`cmd+H` Hide is reserved system-wide).

- **D-12:** **`setup.sh` banner integration = brief one-liner.** Step 7 banner (after the existing `Local voice dictation. Nothing leaves your Mac.` tagline) gets one additional line:
  > `HUD options: PURPLEVOICE_HUD_OFF=1 to disable, PURPLEVOICE_HUD_POSITION=top-right (etc) for alt placement`
  Discoverable, non-intrusive. Full documentation goes in `README.md` (extending the existing `## Security & Privacy` section or a new `## HUD Configuration` subsection).

### Screen-Recording Visibility

- **D-13:** **HUD hidden from screen recordings by default.** Use `NSWindowSharingNone` (or the Hammerspoon `hs.canvas:behavior(...)` / `hs.window.setBehaviorByLabels(...)` equivalent — researcher to identify the exact API surface). Excluded from `screencapture`, QuickTime recording, OBS, Zoom share-screen, etc. Aligned with the `Local voice dictation. Nothing leaves your Mac.` ethos: a journalist demoing PurpleVoice in a recorded video shouldn't have a `● Recording` indicator burned into every demo capture.

- **D-14:** **Honest caveat in README + SECURITY.md.** Add a paragraph to README's `## Security & Privacy` section AND a sentence-level acknowledgement somewhere in SECURITY.md (likely §"How to Verify These Claims" or §"Air-Gapped Installation"):
  > "PurpleVoice attempts to hide the HUD from screen recordings via `NSWindowSharingNone`. macOS-version-specific behaviour may cause the HUD to appear in some capture tools or recording paths. If absolute privacy is required for a recording session, run with `PURPLEVOICE_HUD_OFF=1`."
  Matches SECURITY.md's "auditable + verifiable, not compliant + certified" tone. Best-effort framing — never "guarantees".

- **D-15:** **No `PURPLEVOICE_HUD_VISIBLE_IN_RECORDINGS` env var in v1.** The privacy-first default is the only mode shipped. Users who want the HUD in their recordings can post-process or run with HUD off + use OBS/QuickTime indicator overlays. Adding a third env var to flip the privacy default would dilute the simplicity of the two-knob design (D-10).

### Tagline Integration

- **D-16:** **Tagline `Local voice dictation. Nothing leaves your Mac.` does NOT appear in the HUD.** Phase 2.5 D-12 reserved the tagline for "the about-screen / tooltip" — but the HUD is non-interactive (no hover surface; only visible during press-hold while user's hand is on the hotkey). The tagline already lives in the README header, the setup.sh banner, and the Hammerspoon module load alert (line 339 of `purplevoice-lua/init.lua`). Adding it to the HUD would clutter a transient indicator. Tagline integration deferred to v2 (e.g., right-click menu or About-style popup) if the HUD ever becomes interactive.

### Claude's Discretion

- **Exact corner radius** — suggest half-height (~18px) for cleanly-rounded pill; planner can tune within 14–20px.
- **Font weight + size** — system bold ~13pt likely; planner picks based on macOS UI text conventions.
- **Animation timing** — within HUD-01 (50ms appear, 250ms disappear). Planner picks instant vs 100ms fade-in based on RESEARCH ergonomics findings.
- **Exact alpha value** — ~0.85 stated, planner can tune within 0.80–0.90 if rendering on Sequoia 15.7.5 needs adjustment.
- **Window level / z-order** — above all normal windows but excluded from screen captures. Researcher to identify the right `hs.canvas:level()` / `behavior` combination.
- **API choice (`hs.canvas` vs `hs.drawing` vs `hs._asm.uitk` vs others)** — RESEARCH flag in ROADMAP §"Phase 3.5" specifically calls this out: "`hs.canvas` vs `hs.drawing` ergonomics for a low-CPU 'show only on press' overlay; whether a transparent always-on-top window has any focus-stealing implications." Researcher decides; planner reflects choice in PLAN.
- **Dark-mode handling** — macOS appearance change behaviour. White text on lavender stays readable in both light and dark modes; researcher confirms no adjustment needed.
- **First-load smoke test approach** — manual walkthrough (`tests/manual/test_hud.md`) vs an automated screenshot-pixel-check vs visual sanity-check during install. Planner picks based on RESEARCH automation feasibility.
- **Screen-recording-test verification** — should `tests/security/verify_egress.sh` or a new `tests/security/verify_hud_invisible_in_capture.sh` script attempt a programmatic screencapture and assert the HUD is absent? Planner decides; if too brittle (Apple changes APIs), document as Manual-Only Verification in VALIDATION.md instead.

<specifics>
## Specific Ideas

- **The HUD is an indicator, not a widget.** It surfaces state during press-hold. It's not interactive (no hover, no click, no menu). This shapes every decision: no tooltip, no drag-to-reposition, no right-click menu, no animation other than appear/disappear.

- **Pill shape over dot-only.** The menubar indicator is the dot. The HUD adds the word `Recording` because it appears alone (no menubar context immediately adjacent — top-center positioning specifically separates it from the menubar). Two-token comprehension matters when the user's eyes catch it in peripheral vision.

- **Translucent without blur is a deliberate compromise.** The user explicitly chose modern semi-transparent appearance but rejected the GPU-heavy backdrop blur. Result: alpha ~0.85 lavender rectangle, no vibrancy / blur layer. Rendering cost stays low; no text-edge artifacts; visual feels modern.

- **Top-center on active screen is the "where the user looks when they speak" position.** Cursor-following was rejected (covers content, awkward when typing). Top-right was rejected (collides with notifications dropdown). Bottom-center was rejected (Dock collision risk). Top-center + active-screen is the cleanest compromise.

- **Two env vars only — `PURPLEVOICE_HUD_OFF` + `PURPLEVOICE_HUD_POSITION`.** No alpha, no colour, no offset, no font-size knobs. The brand is locked (Phase 2.5 D-09); exposing more visual knobs invites brand drift. Power users who want different visuals can fork.

- **Hidden from screen recordings is a privacy claim that aligns with Phase 2.7 SECURITY.md.** A journalist demoing PurpleVoice in a recorded video doesn't accidentally watermark every capture with a `● Recording` indicator. The framing in README + SECURITY.md is "best-effort" — same honesty discipline as Phase 2.7's egress / SBOM / reproducibility caveats. Never claim "guarantees absolute privacy"; claim "attempts to hide; macOS-version behaviour may vary".

- **Default-on with opt-out (`PURPLEVOICE_HUD_OFF=1`) matches `PURPLEVOICE_NO_SOUNDS` precedent.** First-run users discover the HUD organically; anyone who finds it distracting flips one env var and reloads Hammerspoon.

- **The tagline is too valuable to clutter a 140×36 pill that appears every few seconds.** Phase 2.5 D-12 reserved the tagline for an "about-screen / tooltip" — but the HUD is non-interactive in v1 (no surface for tooltip; press-hold means user's hand is occupied). Tagline stays in README header / setup.sh banner / load alert; HUD-tagline integration deferred to v2 if the HUD ever becomes interactive (e.g., right-click menu).

</specifics>

<deferred>
## Deferred Ideas

- **Runtime toggle hotkey** (e.g., `cmd+option+shift+h` to show/hide HUD without Hammerspoon reload). Adds runtime state + collision-risk surface; rejected for v1 in favour of env-var-only config.
- **Alpha override env var** (`PURPLEVOICE_HUD_ALPHA=0.85`). Rejected — two knobs is enough for v1.
- **Colour override env var** (`PURPLEVOICE_HUD_COLOR=#ABCDEF`). Rejected — contradicts Phase 2.5 brand lock.
- **Coordinate-precise position override** (`PURPLEVOICE_HUD_X` / `PURPLEVOICE_HUD_Y`). Rejected — six named positions cover the realistic preference space; explicit coords add screen-dimension-arithmetic burden on the user.
- **Tagline integration in HUD** — deferred to v2 if the HUD becomes interactive (right-click menu, About-style popup). Tagline currently lives in README header + setup.sh banner + Hammerspoon load alert.
- **Auto-detect screen-recording API failures + warn user** — rejected; honest caveat in README + SECURITY.md is sufficient. If Apple changes NSWindow sharingType behaviour, the caveat covers it without runtime detection complexity.
- **Visible-in-recordings env var override** (`PURPLEVOICE_HUD_VISIBLE_IN_RECORDINGS=1`) — rejected; the privacy-first default is the only mode shipped in v1.
- **Dark-mode-specific colour palette** — likely no adjustment needed (white-on-lavender stays readable in both modes); researcher confirms.
- **Idle-state visual** (subtle dim presence vs hidden when idle) — rejected; D-09 / HUD-03 specify hidden when idle for zero-CPU + zero-distraction.
- **Multi-line content** (e.g., `Recording` + a short subtitle) — rejected; pill stays compact 140×36, single-line content.
- **Glassmorphism with backdrop blur** — explicitly rejected by user ("option 3 without the blur"); avoids GPU cost + text-edge artifacts.
- **Comprehensive visual property env vars** — rejected; brand drift concern.
- **Drag-to-reposition with persistence** — rejected; non-interactive surface; named positions cover the use case.
- **First-run tutorial overlay** ("Press cmd+shift+e to record") — rejected; PurpleVoice is single-user (Oliver) and the README + setup.sh banner already document the hotkey.
