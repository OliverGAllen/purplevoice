# Phase 3.5: Hover UI / HUD — Research

**Researched:** 2026-04-29
**Domain:** macOS overlay UI via Hammerspoon (`hs.canvas` API on Sequoia 15.7.5)
**Confidence:** HIGH (Hammerspoon API verified against upstream source; macOS sharingType behaviour verified against Apple Developer Forums + community reports)

***

## Phase Context Summary

Phase 3.5 layers a small floating "● Recording" pill on top of the existing PurpleVoice push-to-talk loop. The shape is locked (140×36 lavender rounded-rectangle, alpha ~0.85, no blur), the lifecycle is locked (visible while hotkey held; hidden otherwise), the configuration surface is locked (two env vars `PURPLEVOICE_HUD_OFF` + `PURPLEVOICE_HUD_POSITION`; no runtime hotkey), and the implementation point is locked (`purplevoice-lua/init.lua`, hooked into existing `onPress` / `resetState`). The user has gathered 16 decisions in `03.5-CONTEXT.md`; this document fills in the technical depth needed to plan around those decisions, not to revisit them.

**Two findings drive the most consequential planning shifts** versus the discussion-stage assumptions:

1. **`hs.canvas` is the correct API.** `hs.drawing` is officially deprecated and slated for removal; `hs.alert` (which the codebase already uses for the load message) still uses `hs.drawing` internally but that's a Hammerspoon legacy detail the planner should not inherit. All new code in this phase should use `hs.canvas`.
2. **The "hidden from screen recordings" claim (D-13) needs re-framing.** `NSWindowSharingNone` (the AppKit constant referenced in `03.5-CONTEXT.md`) is **broken on macOS 15+ for ScreenCaptureKit-using capture tools (OBS, QuickTime, Zoom share-screen)**. Apple confirmed via Developer Forums there is currently no public API to exclude a window from capture on macOS 15+. Worse, `hs.canvas` does not expose `setSharingType` at all — the canvas's underlying `HSCanvasWindow` (subclass of `NSWindow`) does not call `setSharingType:NSWindowSharingNone` anywhere in Hammerspoon's source. This means **(a)** even a clean implementation could only hit the legacy CoreGraphics `screencapture` CLI path on Sequoia; **(b)** the ScreenCaptureKit path (OBS, modern QuickTime, Zoom share-screen) will capture the HUD regardless. The D-14 caveat needs to be more honest than "best-effort" — closer to "may be honoured by legacy `screencapture` CLI on macOS 14 and below; will NOT be honoured by ScreenCaptureKit-based tools (OBS, modern QuickTime, Zoom share-screen) on macOS 15+. If absolute privacy is required, run with `PURPLEVOICE_HUD_OFF=1`." See Priority 2 below.

**Primary recommendation:** Build the HUD as a single `hs.canvas` instance created at module load (hidden), with `level("status")`, `behavior({"canJoinAllSpaces", "stationary"})`, `wantsLayer(true)`, `:alpha(0.85)`. Show on press with `:show(0)` (instant — fade is unnecessary visual noise for a press-hold indicator). Hide on release with `:hide(0.15)` (matches `hs.alert.defaultStyle.fadeOutDuration`; the user perceives this as "the indicator just disappeared" while the cleanup animation runs). Do NOT pursue `setSharingType:NSWindowSharingNone` — it is not exposed by `hs.canvas`, and even if accessed via the unofficial `hs._asm.objc` bridge, it does not work on macOS 15+ for the capture tools that matter. Document the screen-recording caveat honestly per Phase 2.7 D-17 framing discipline (drop "best-effort" → "limited; see SECURITY.md").

## State of the Art (12 Research Priorities)

### Priority 1: `hs.canvas` vs `hs.drawing` ergonomics

**Confidence:** HIGH. Verified against Hammerspoon official docs + `libcanvas.m` source.

**Decision:** Use `hs.canvas`. `hs.drawing` is officially deprecated ("hs.drawing is now deprecated and will be removed in a future release, with its functionality now implemented by hs.canvas") and Hammerspoon's own `hs.alert` module uses the older `hs.drawing` only as a legacy artifact. New code should use `hs.canvas`.

**Why `hs.canvas` is the right shape for this HUD:**
- Single declarative element table — describe the rectangle, the text, attributes once.
- Separate `:show(fadeIn)` / `:hide(fadeOut)` lifecycle methods that animate alpha; perfect for press/release.
- `wantsLayer(true)` enables Core Animation rendering — GPU-composited, low CPU.
- `level()` and `behavior()` give precise window-stacking control without going to ObjC.

**Element types you need:**

| Element type | Purpose | Key attributes |
|--------------|---------|----------------|
| `"rectangle"` | The lavender pill | `frame`, `roundedRectRadii = { xRadius = 18, yRadius = 18 }`, `fillColor`, `strokeColor`, `action = "fill"` (or `"strokeAndFill"`) |
| `"text"` | The "● Recording" label overlaid on the pill | `frame`, `text` (string OR `hs.styledtext`), `textColor`, `textFont`, `textSize`, `textAlignment = "center"` |

**Source:** [Hammerspoon hs.canvas docs](https://www.hammerspoon.org/docs/hs.canvas.html); [libcanvas.m source](https://github.com/Hammerspoon/hammerspoon/blob/master/extensions/canvas/libcanvas.m); [canvas.lua source](https://github.com/Hammerspoon/hammerspoon/blob/master/extensions/canvas/canvas.lua).

### Priority 2: Hiding from screen recordings (D-13) — **CRITICAL FRAMING ISSUE**

**Confidence:** HIGH (limitation confirmed by Apple Developer Forums + Tauri issue + multiple developer reports).

**The bottom line: `hs.canvas` does NOT expose `setSharingType:NSWindowSharingNone`, AND even if it did, that flag is broken on macOS 15+ for ScreenCaptureKit-using tools (OBS, modern QuickTime, Zoom share-screen).**

#### What was verified

1. **Hammerspoon source audit** (`extensions/canvas/libcanvas.m`): The `HSCanvasWindow` class (subclass of `NSWindow`) sets these properties at init: `backgroundColor = clearColor`, `opaque = NO`, `hasShadow = NO`, `ignoresMouseEvents = YES`, `restorable = NO`, `hidesOnDeactivate = NO`, `animationBehavior = NSWindowAnimationBehaviorNone`, `level = NSScreenSaverWindowLevel`. **It does NOT call `setSharingType:NSWindowSharingNone` anywhere.** So out-of-the-box, the HUD canvas has the *default* `sharingType = NSWindowSharingReadOnly`, meaning legacy CoreGraphics capture tools CAN capture it.
2. **Apple Developer Forums (thread 792152, 2025):** "On macOS 15.4+, NSWindow with sharingType = .none results in the window being captured by ScreenCaptureKit." Apple Developer Technical Support engineer confirmed: *"At this time there are no public APIs for preventing screen capture."*
3. **Tauri issue #14200:** "macOS 15+: ScreenCaptureKit ignores setContentProtection / NSWindow.sharingType. This is a deliberate change by Apple and currently has no known workaround."
4. **Practical reality on macOS Sequoia 15.7.5 (Oliver's machine):** ScreenCaptureKit-using capture tools include modern QuickTime screen recording, OBS Studio, Zoom share-screen, Discord screen-share, Loom — all major modern tools. Only the legacy `screencapture` CLI (`/usr/sbin/screencapture`) might still honour `kCGWindowSharingNone` on macOS 15, and even that is unverified for 15.7+.

#### The honest framing for D-14

The `03.5-CONTEXT.md` D-14 caveat as currently drafted is **insufficient**:

> "PurpleVoice attempts to hide the HUD from screen recordings via `NSWindowSharingNone`. macOS-version-specific behaviour may cause the HUD to appear in some capture tools or recording paths. If absolute privacy is required for a recording session, run with `PURPLEVOICE_HUD_OFF=1`."

This understates the issue. A journalist on macOS 15.7.5 demoing PurpleVoice on Zoom share-screen will see the HUD in their recording — guaranteed. The "may cause" framing implies it's an edge case; in fact it's the common case for ScreenCaptureKit-using tools on Sequoia and later.

**Recommended honest framing for the planner to adopt** (consistent with Phase 2.7 D-17 "auditable + verifiable" discipline):

> "PurpleVoice attempts to hide the HUD from screen recordings, but this is **limited** on modern macOS. The legacy `NSWindowSharingNone` flag is honoured by older CoreGraphics-based capture tools (the macOS `screencapture` command-line on macOS 14 and below) but is **ignored by ScreenCaptureKit on macOS 15 and above** — meaning OBS, Zoom share-screen, modern QuickTime, Discord, Loom, and similar tools will capture the HUD regardless. Apple has stated there are no public APIs to prevent ScreenCaptureKit capture (Apple Developer Forums thread 792152, 2025). If absolute privacy is required for a recording session, run with `PURPLEVOICE_HUD_OFF=1` and rely on the menubar indicator alone."

This framing **passes** the Phase 2.7 framing lint (`tests/test_security_md_framing.sh`):
- Does NOT use `compliant` / `certified` / `guarantees`.
- Uses "limited" / "honoured" / "ignored" / "stated" — all qualified.
- Cross-references the authoritative source (Apple's own forum).

**Recommended approach for the implementation:**

| Question | Answer | Rationale |
|----------|--------|-----------|
| Should we attempt to set `sharingType` on the canvas window? | **No, in v1.** | `hs.canvas` doesn't expose it; the unofficial `hs._asm.objc` bridge would add a runtime dependency for marginal benefit (legacy tools only on macOS ≤ 14). Cost > value. |
| Should we set `setBehaviorByLabels({"transient", ...})` to weakly hint exclusion? | **Yes, harmless.** | `transient` doesn't actually hide from capture, but it does hint to AppKit that the window is ephemeral; combined with `stationary` it ensures consistent space behaviour. |
| Should we re-frame D-14 as I've drafted above? | **Yes — load-bearing for trust narrative.** | A "best-effort" claim that a Zoom share-screen demo demonstrably violates would damage SECURITY.md credibility. Phase 2.7 D-17 demands "compatible with" framing; this is the same discipline applied to the HUD. |
| Should the HUD code path try to detect ScreenCaptureKit usage and warn? | **No.** Already deferred per `03.5-CONTEXT.md` "Reviewed Todos". | Auto-detection is over-complex; a clear caveat in the docs is sufficient. |

**Sources:**
- [Apple Developer Forums thread 792152: macOS 15.4+ NSWindow sharingType](https://developer.apple.com/forums/thread/792152) — Apple DTS confirms no public API.
- [Tauri Issue #14200: macOS 15+ ScreenCaptureKit ignores NSWindow.sharingType](https://github.com/tauri-apps/tauri/issues/14200) — community confirmation.

### Priority 3: Window level for "above all, no focus steal"

**Confidence:** HIGH. Verified against `libcanvas.m` source and `hs.canvas.windowLevels` enumeration.

**Decision:** Use `level("status")` or accept the default `level("screenSaver")`. Either works; `"status"` is semantically more accurate (this is a status indicator, not a screensaver), so it's the preferred name. Both are above normal app windows.

**Why focus is not stolen — it's already handled by `hs.canvas`:**

From the `libcanvas.m` source, `HSCanvasWindow` sets these defaults at init (no Lua-side configuration required):
```objc
self.ignoresMouseEvents = YES;       // mouse clicks pass through
self.hidesOnDeactivate  = NO;        // doesn't disappear when user clicks away
self.animationBehavior  = NSWindowAnimationBehaviorNone;
self.level              = NSScreenSaverWindowLevel;
```

And the override:
```objc
- (BOOL)canBecomeKeyWindow {
    // returns YES only if a child canvas element has explicitly opted in via canBecomeKeyView
    // the HUD has no input elements, so this returns NO → no focus stealing
}
```

So as long as the HUD canvas has only `"rectangle"` and `"text"` elements (no text input, no canvas-elements that override `canBecomeKeyView`), focus is never stolen.

**Behavior labels to set:**

| Label | Why | Source |
|-------|-----|--------|
| `canJoinAllSpaces` | HUD appears across virtual desktops (Spaces) without re-creation. User on Space 1 holds hotkey, switches to Space 2 mid-record (rare but possible) — HUD follows. | `hs.canvas.windowBehaviors` enumeration |
| `stationary` | HUD doesn't move to active Space when user switches Spaces — already covered by `canJoinAllSpaces` but `stationary` makes the intent explicit. | `hs.canvas.windowBehaviors` |
| `transient` | Hint to AppKit that the window is ephemeral; excluded from window-cycling (`cmd+~`). Cheap, no downside. | `hs.canvas.windowBehaviors` |
| (DO NOT use) `ignoresCycle` | Subset of `transient`; pick `transient` for clearer semantics. | — |
| (DO NOT use) `moveToActiveSpace` | Conflicts with `canJoinAllSpaces`. | — |

Final call: `behaviorAsLabels({"canJoinAllSpaces", "stationary", "transient"})`.

**Important: `hs.canvas:behavior()` flag handling.** When passing labels:
- A **string** argument *toggles* that label.
- A **table** argument *replaces* behavior with the union of the labels.

So always use the table form to be deterministic: `:behavior({"canJoinAllSpaces", "stationary", "transient"})`. Or use the helper `:behaviorAsLabels({"canJoinAllSpaces", "stationary", "transient"})`.

**Sources:**
- [libcanvas.m HSCanvasWindow](https://github.com/Hammerspoon/hammerspoon/blob/master/extensions/canvas/libcanvas.m)
- [hs.canvas.windowLevels](https://www.hammerspoon.org/docs/hs.canvas.html)

### Priority 4: Show/hide animation timing

**Confidence:** HIGH. Verified against `libcanvas.m` `fadeIn:`/`fadeOut:` methods and `hs.alert` reference values.

**Decision:** `:show(0)` (instant) on press, `:hide(0.15)` on release.

**Why instant on show:** The user is **deliberately** holding the hotkey — they're paying attention. A fade-in (any duration > 0) introduces visual lag that feels like the HUD is "waking up". For a press-hold indicator, instant feedback aligns with the press-hold mental model. Hammerspoon's own `hs.alert` uses a 150ms fade-in for *unsolicited* alerts (which arrive without the user's intent and benefit from a "soft entrance"); a press-hold indicator is solicited and benefits from immediacy.

**Why 150ms on hide:** The user just stopped paying attention to the HUD (released the key). A short fade-out smooths the transition without delaying the "did it work?" signal. 150ms matches `hs.alert.defaultStyle.fadeOutDuration` (the canonical Hammerspoon overlay fade). Well within HUD-01's 250ms budget.

**API signatures (verified from `libcanvas.m`):**

```objc
- (void)fadeIn:(NSTimeInterval)fadeTime {
    CGFloat alphaSetting = self.alphaValue ;
    [self setAlphaValue:0.0];
    [self makeKeyAndOrderFront:nil];
    [NSAnimationContext beginGrouping];
      [[NSAnimationContext currentContext] setDuration:fadeTime];
      [[self animator] setAlphaValue:alphaSetting];
    [NSAnimationContext endGrouping];
}
```

So `:show(0)` skips the NSAnimationContext block (Lua-side, `fadeTime == 0` → no animation). `:show(0.15)` would fade alpha from 0 → 0.85 over 150ms.

`:hide(fadeTime)` is symmetric: alpha fades 0.85 → 0 over fadeTime.

**Critical pitfall (Pitfall 4 below):** if the user releases within < 50ms of pressing (rapid-tap), `:show(0)` then `:hide(0.15)` happens almost instantly — the HUD flashes briefly. This is fine and even desirable (gives feedback that the rapid-tap was registered). Existing `tests/test_duration_gate.sh` covers the audio-pipeline side; the HUD shouldn't introduce new gating logic for this.

**Lower bound on fade durations:** Empirically, NSAnimationContext durations below ~16ms (one 60Hz frame) collapse to instant. So `0.0` and `0.01` are equivalent in practice. Use `0` for clarity.

**Sources:**
- [libcanvas.m fadeIn: / fadeOut:](https://github.com/Hammerspoon/hammerspoon/blob/master/extensions/canvas/libcanvas.m)
- [hs.alert.lua defaultStyle](https://github.com/Hammerspoon/hammerspoon/blob/master/extensions/alert/alert.lua) (fadeIn/fadeOut both default to 0.15)

### Priority 5: Multi-monitor active-screen resolution

**Confidence:** HIGH. Verified against `hs.screen` docs.

**Decision:** Resolve the active screen each time `onPress` fires. Re-position the canvas before showing.

**Resolution algorithm (sequential fallback):**

```lua
local function activeScreen()
  local fw = hs.window.focusedWindow()
  if fw then return fw:screen() end          -- (1) focused window's screen
  return hs.screen.mainScreen()              -- (2) macOS "main" (containing focused window if any) → effectively the same
                                             --    or the screen the menubar is on if no focus
                                             -- mainScreen() never returns nil.
end
```

**Why per-press resolution:** The user might move their focused window to a different monitor between presses. Caching the screen at module load → HUD appears on the wrong screen after a multi-monitor reshuffle. Per-press resolution costs one Lua call (~microseconds) and is correct.

**Edge case (no focused window):** `hs.window.focusedWindow()` can return nil if the desktop has focus (e.g., after an app quit or before the first focus event). `hs.screen.mainScreen()` is the documented fallback — it returns "the screen containing the currently focused window, or the screen with the menubar if no window is focused". It never returns nil on a system with at least one display.

**`primaryScreen()` vs `mainScreen()`:** **`mainScreen()` is what you want.** `primaryScreen()` is the user's System Settings "Primary" screen (the one with the menubar in single-display fashion); on a multi-monitor setup it's a fixed reference, not the "where the user is working" screen.

**Frame coordinates:**
- `hs.screen:frame()` excludes menubar + dock (usable area). Returns `{x, y, w, h}` in absolute global coordinates.
- `hs.screen:fullFrame()` includes menubar + dock. Returns `{x, y, w, h}` in absolute global coordinates.

For positioning the HUD `~50px below the menubar`, use `:fullFrame()` and add 50: the menubar is ~25px tall, so a 50px offset from `fullFrame.y` lands the HUD ~25px below the menubar (cleanly clear of it). Alternatively, use `:frame().y` directly (which already accounts for the menubar) and add a smaller margin like 25px. **Recommendation:** use `:fullFrame()` + 50 — explicit and readable.

**Multi-monitor coordinate origin:** macOS uses absolute global coordinates. Each screen has its own `(x, y, w, h)` in this global space, and the screens may have arbitrary offsets relative to each other (left, right, above, below — depending on user's Display Arrangement). The position arithmetic in Priority 6 assumes screen-local thinking but uses `:fullFrame()` values which are global, so you don't need a separate "translate to screen origin" step — `screenFrame.x` and `screenFrame.y` ARE the global origin.

**Sources:**
- [hs.screen docs](https://www.hammerspoon.org/docs/hs.screen.html)
- [hs.alert.lua positioning](https://github.com/Hammerspoon/hammerspoon/blob/master/extensions/alert/alert.lua) (uses `screenFrame.x + (screenFrame.w - drawingFrame.w) / 2` — same idiom)

### Priority 6: Position arithmetic for the six named positions

**Confidence:** HIGH. Standard rect-arithmetic; verified pattern from `hs.alert.lua`.

**The six named positions:**

```lua
-- Constants
local HUD_W = 140
local HUD_H = 36
local EDGE_MARGIN = 24      -- standard macOS edge padding (e.g., Notification Center panel uses ~20-24px)
local TOP_GAP = 50          -- D-08: ~50px below menubar
local NEAR_CURSOR_OFFSET = 30   -- offset below cursor

local function positionFor(name, screenFrame)
  -- screenFrame: hs.screen:fullFrame() — table with x, y, w, h in global coords
  if name == "top-center" then
    return {
      x = screenFrame.x + (screenFrame.w - HUD_W) / 2,
      y = screenFrame.y + TOP_GAP,
    }
  elseif name == "top-right" then
    return {
      x = screenFrame.x + screenFrame.w - HUD_W - EDGE_MARGIN,
      y = screenFrame.y + TOP_GAP,
    }
  elseif name == "bottom-center" then
    return {
      x = screenFrame.x + (screenFrame.w - HUD_W) / 2,
      y = screenFrame.y + screenFrame.h - HUD_H - EDGE_MARGIN,
    }
  elseif name == "bottom-right" then
    return {
      x = screenFrame.x + screenFrame.w - HUD_W - EDGE_MARGIN,
      y = screenFrame.y + screenFrame.h - HUD_H - EDGE_MARGIN,
    }
  elseif name == "near-cursor" then
    local mp = hs.mouse.absolutePosition()  -- global coords
    return {
      x = mp.x - HUD_W / 2,
      y = mp.y + NEAR_CURSOR_OFFSET,        -- below cursor; doesn't cover the cursor itself
    }
  elseif name == "center" then
    return {
      x = screenFrame.x + (screenFrame.w - HUD_W) / 2,
      y = screenFrame.y + (screenFrame.h - HUD_H) / 2,
    }
  else
    -- Invalid: fall back to top-center per D-07
    return {
      x = screenFrame.x + (screenFrame.w - HUD_W) / 2,
      y = screenFrame.y + TOP_GAP,
    }
  end
end
```

**Margin recommendations (researcher's call per `<decisions>` Claude's Discretion):**
- **`EDGE_MARGIN = 24`** — matches macOS Notification Center pane edge gap. Visually balanced for top-right / bottom-right / bottom-center.
- **`TOP_GAP = 50`** — D-08 directly. ~25px menubar + 25px breathing room.
- **`NEAR_CURSOR_OFFSET = 30`** — below the cursor; doesn't occlude the cursor; far enough that the user's hand on the mouse doesn't block the indicator visually.

**Notes on `near-cursor`:**
- `hs.mouse.absolutePosition()` returns global coords (across all displays). No screen-local translation needed.
- If the cursor is near the bottom of a screen, `near-cursor` could push the HUD off-screen. **Recommendation:** clamp y to `screenFrame.y + screenFrame.h - HUD_H - EDGE_MARGIN` after computing. Optional polish; can defer if the planner wants minimal v1.
- The cursor screen may differ from the focused-window screen. For `near-cursor`, use the *cursor's* screen (`hs.mouse.getCurrentScreen()` if it exists, otherwise `hs.screen.find(hs.mouse.absolutePosition())`); for the other five named positions, use the focused-window screen per D-06.

**Cursor screen API:** `hs.mouse.getCurrentScreen()` is documented but not consistently across versions. Safer pattern: `hs.screen.find(hs.mouse.absolutePosition())` — works on all versions, returns the screen containing the point.

### Priority 7: Reload-friendly module-load pattern

**Confidence:** HIGH. Verified against existing `purplevoice-lua/init.lua` patterns (menubar lifecycle, env var reads, `pcall`-wrapped cleanup).

**Decision:** Mirror the existing `menubar = hs.menubar.new()` pattern (line 77 of `init.lua`). Hammerspoon's reload behaviour does NOT explicitly clean up module-level userdata; instead, garbage collection handles orphaned canvases when the Lua state is torn down. **However**, the safer pattern is to delete-if-exists at the top of the new module load to defend against partial-reload edge cases.

**Reload-friendly pattern:**

```lua
-- ----------------------------------------------------------------
-- HUD canvas (Phase 3.5; defence-in-depth cleanup on reload)
-- ----------------------------------------------------------------
-- Hammerspoon reloads init.lua on every config change. Module-level
-- variables become eligible for GC, and Hammerspoon's framework GC-runs
-- the previous canvas when the Lua state is collected. As a defence-in-depth
-- measure, if a module-global "_purplevoice_hud" canvas is still around at
-- module load, delete it explicitly before creating the new one. This
-- closes a hypothetical orphan-canvas window leak on partial-reload paths.
if _G._purplevoice_hud then
  pcall(function() _G._purplevoice_hud:delete() end)
  _G._purplevoice_hud = nil
end

local hudCanvas = nil
if hudEnabled then
  hudCanvas = hs.canvas.new({ x = 0, y = 0, w = HUD_W, h = HUD_H })
    :level("status")
    :behaviorAsLabels({"canJoinAllSpaces", "stationary", "transient"})
    :wantsLayer(true)
    :alpha(0)  -- start fully transparent; alpha set to HUD_ALPHA on show
  -- ... append elements ...
  _G._purplevoice_hud = hudCanvas  -- expose for next-reload cleanup
end
```

**Why `_G._purplevoice_hud` (a global) instead of a local?**

Locals get GC'd on reload — fine for the current process but defensive cleanup needs to *find* the previous instance. A name-spaced global (`_G._purplevoice_hud`, distinct from `M.hud` which is module-internal) survives across the reload boundary because Hammerspoon's Lua state persists during a `hs.reload()`. This is **the same pattern used by the `pcall(hs.notify.unregister, "voiceccOpenMicSettings")` block at line 26 of the existing `init.lua`** — defending against orphaned tags from previous module versions.

**Alternative considered:** Don't bother with the global; rely on GC. The risk is negligible in practice (Hammerspoon's GC handles canvas userdata correctly), but the explicit cleanup is cheap and matches the existing `pcall(hs.notify.unregister, ...)` defensive pattern. **Recommendation:** include the cleanup block — costs 4 lines, eliminates a class of edge-case bugs, signals discipline to future maintainers.

**Why `hs.menubar.new()` (line 77) doesn't need the same defensive cleanup:** `hs.menubar` returns an idiomatic userdata that the framework auto-tracks; menubar items get torn down by the framework on reload. Canvas instances are slightly different — they wrap an `NSWindow`, and explicit `:delete()` releases the window cleanly without waiting for GC. The defensive cleanup is more important for canvas than for menubar.

**Source:** [purplevoice-lua/init.lua line 26 (existing `pcall(hs.notify.unregister)` precedent)](file:///Users/oliverallen/Temp%20video/voice-cc/purplevoice-lua/init.lua); [hs.canvas:delete()](https://www.hammerspoon.org/docs/hs.canvas.html).

### Priority 8: Performance target — "zero CPU when idle"

**Confidence:** HIGH for `hs.canvas:hide()` semantics; MEDIUM-HIGH for empirical CPU baseline (`top -pid <pid>` testing recommended at task-execution time).

**Decision:** Create the canvas once at module load (hidden); `:show(0)` / `:hide(0.15)` on the press/release lifecycle. Do NOT delete-and-recreate on each press.

**Why this hits zero CPU when idle:**
1. A hidden `hs.canvas` calls `[NSWindow orderOut:nil]` (verified from `libcanvas.m`'s show/hide implementation) — the NSWindow is removed from the window list, the WindowServer no longer composites it, no draw calls happen.
2. `wantsLayer(true)` causes Core Animation to handle the visible-state composite on the GPU; CPU side is idle.
3. There's no animation loop, no timer, no event tap. Once hidden, the canvas is inert.

**What "idle CPU" looks like on Hammerspoon's process:** Hammerspoon's main process holds many long-lived resources (event taps, menubar items, hotkey bindings, pasteboard observers). Any HUD-related CPU consumption is a tiny fraction of the baseline. Verifying "zero" empirically requires:
1. Note Hammerspoon's idle-state CPU (e.g., 0.1–0.3% on M2 Pro): `top -pid $(pgrep Hammerspoon)`
2. Load the HUD module, leave HUD hidden.
3. Measure again — should be within noise (±0.1%).
4. Trigger HUD show/hide cycle 100x with `:show(0); hs.timer.usleep(100000); :hide(0.15)`.
5. Measure idle CPU after the cycle ends — should return to baseline.

**Show vs delete-and-recreate tradeoff:**
| Approach | CPU when idle | Latency on press | Memory | Recommended |
|----------|---------------|------------------|--------|-------------|
| Create-once + `:show()/:hide()` | Zero (hidden NSWindow not composited) | Instant (orderFront) | ~few hundred bytes for canvas userdata + NSWindow | **YES** |
| Delete-and-recreate | Zero | ~5-20ms (canvas allocation + NSWindow construction) | Zero between presses | No — adds latency, no benefit |

**Hammerspoon canvas behavior (verified):** `:hide()` orders the window out, `:show()` orders it back in. The canvas's element list (rectangle + text) is preserved across hide/show; you don't need to re-append elements every press.

**`:isShowing()` for state queries:** Returns a bool indicating whether the canvas was last `:show()`-ed (not whether it's currently *visible* — that's `:isOccluded()`). Useful for defensive programming: `if hudCanvas and not hudCanvas:isShowing() then hudCanvas:show(0) end`.

**Reduced-redraw pattern:** Don't mutate the canvas elements per-press. The pill geometry (140×36 lavender rounded-rect with "● Recording" centered text) is invariant — set it up once at module load, then just `:show()` / `:hide()` / `:topLeft({x, y})` on each press cycle.

### Priority 11: Dark-mode handling

**Confidence:** MEDIUM. Empirical verification recommended at execution time.

**Decision (subject to live verification):** White text on lavender (#B388EB at α=0.85) is readable in both light and dark modes; **no adjustment expected**. The lavender background carries enough chroma (HSL 261°, 71%, 73%) that white text contrasts adequately regardless of system appearance. The 0.85 alpha lets some background colour bleed through, but with the rounded-rect fillColor = solid lavender, the *text* sits on near-fully-opaque lavender (the `roundedRectRadii` rectangle is opaque-fill-with-window-alpha, not per-pixel alpha — so the text is on lavender, not on a translucent ghost of the desktop).

**What to verify in implementation:**

- Most likely fix: bump alpha 0.85 → 0.90 (more opaque background = less interference from underlying pixels).
- Less likely: switch text colour to a slightly off-white (e.g., `#F5F5F5`) for less harsh contrast in light mode. **Don't** add a darkmode/lightmode env var; just pick a single colour that works in both.

**No `hs.host.appearance` / OS-appearance-callback wiring needed.** The HUD is press-hold transient (visible for ~1-5 seconds at a time); the OS appearance changes happen at sundown / sunrise, not mid-press.

### Priority 12: Exit signaling for HUD lifecycle

**Confidence:** HIGH. Verified against `purplevoice-lua/init.lua` lines 116–124 (`resetState`) and 274–283 (the `onPress`/`hs.task.new` exit callback).

**Decision:** Add `hideHUD()` to `resetState()` alongside `setMenubarIdle()`. Add `showHUD()` to `onPress` alongside `setMenubarRecording()`. **Symmetric with menubar; no new lifecycle layer.**

**The lifecycle is already correct for HUD-04** — every exit code path (0 / 2 / 3 / 10 / 11 / 12) flows through the `pcall(handleExit)` → `resetState()` chain in the `hs.task.new()` callback (lines 274–283). When `purplevoice-record` exits with code 10 (TCC mic deny) or 11 (install incomplete) or 12 (sox/whisper failure), the callback fires, `resetState` runs, `setMenubarIdle()` and `hideHUD()` both execute. The HUD vanishes, the menubar dot returns to outline form, the user sees an actionable notification — symmetric, predictable.

**The integration points (verified from existing source):**

| Existing code (`purplevoice-lua/init.lua`) | Add HUD call | Where |
|--------------------------------------------|--------------|-------|
| Line 271 `setMenubarRecording()` (in `onPress`) | `showHUD()` after the menubar call | onPress, after `setMenubarRecording()` and before `playStartCue()` (or any order — they're independent) |
| Line 123 `setMenubarIdle()` (in `resetState`) | `hideHUD()` before the menubar call | resetState, alongside `setMenubarIdle()` |

**Edge case: re-entrant press during failure cleanup.** `isRecording = false` is set in `resetState()` (line 117) before `setMenubarIdle()`. If a second hotkey press arrives while `resetState` is running, it sees `isRecording = false` (depending on timing) and may enter `onPress` again. **This is fine for the HUD** — `showHUD()` is idempotent (`hs.canvas:show(0)` on an already-visible canvas is a no-op), and the next `resetState` will hide it cleanly.

**Defence-in-depth:** Wrap `showHUD` and `hideHUD` calls in `pcall` to match the existing `pcall(handleExit, ...)` discipline at line 276:

```lua
-- in onPress
local ok = pcall(showHUD)
if not ok then hs.console.printStyledtext("HUD show failed (non-fatal)") end
```

This costs negligible runtime (pcall overhead is microseconds) and means a HUD bug never blocks the recording loop. Matches the existing failsafe-stylings in `init.lua`.

## Common Pitfalls (Phase 3.5 specific)

### Pitfall 1: Treating the HUD as a "best-effort" screen-recording shield

**What goes wrong:** D-14 caveat reads "best-effort", users on macOS Sequoia (15.7.5 = Oliver's machine; 15.4+ = the affected version range) trust it, demo PurpleVoice on a Zoom call with screen-share, and the HUD ends up watermarked into every second of the recording. Trust narrative breaks.

**Why it happens:** `NSWindowSharingNone` was Apple's documented public API for capture exclusion until macOS 15.4. Apple silently broke it for ScreenCaptureKit (the framework used by all modern capture tools — OBS, modern QuickTime, Zoom, Discord, Loom, Teams) and confirmed via Developer Forums that there is no public-API replacement. The legacy CoreGraphics `screencapture` CLI may still honour the flag on macOS 14, but that's a small slice of the capture-tool universe.

**How to avoid:** Re-frame D-14 per the recommended language in Priority 2. Don't pursue `setSharingType` via Lua-ObjC bridge — it's not in `hs.canvas`, and even if it were, it wouldn't help on macOS 15+. Be honest that the HUD is visible in modern screen captures and recommend `PURPLEVOICE_HUD_OFF=1` for sensitive recording sessions.

**Warning signs:** If you find yourself writing "PurpleVoice prevents the HUD from appearing in screen recordings" — STOP. Use "PurpleVoice attempts to limit HUD visibility in legacy screen-capture paths; modern capture tools (ScreenCaptureKit-based) on macOS 15+ will capture the HUD regardless. For sensitive sessions, run with `PURPLEVOICE_HUD_OFF=1`."

### Pitfall 2: Using `hs.drawing` instead of `hs.canvas`

**What goes wrong:** `hs.drawing` is officially deprecated. The codebase already uses `hs.alert` (which is `hs.drawing`-based internally) for the load message — that's *legacy Hammerspoon code* that Phase 3.5 should not extend. New canvas work that uses `hs.drawing` will work today but will break on a future Hammerspoon major-version bump.

**Why it happens:** Old tutorials, Stack Overflow answers, and the existing `hs.alert` pattern in `init.lua` line 339 lead developers toward `hs.drawing`.

**How to avoid:** Use `hs.canvas` for all Phase 3.5 work. The element-table API is more verbose initially but cleaner for the pill-with-text use case (single declarative spec, no `:setStrokeWidth(2):setFillColor(...):setRoundedRectRadii(18, 18):show(0.15)` chain).

### Pitfall 3: Setting the canvas alpha at element-level instead of canvas-level

**What goes wrong:** Setting `fillColor = { hex = "#B388EB", alpha = 0.85 }` on the rectangle element AND setting `:alpha(1)` on the canvas produces a translucent rectangle on a fully-opaque NSWindow → the text on top of the rectangle ends up double-composited (rectangle alpha multiplied with canvas alpha) → text edges look hazy on dark backgrounds.

**Why it happens:** Two places to set alpha; intuitive to set both.

**How to avoid:** Set the rectangle's `fillColor` to alpha 1 (fully opaque), and use `hudCanvas:alpha(0.85)` to set the canvas-level alpha. The canvas alpha is applied by Core Animation as a single composite step over the underlying pixels — text and rectangle composite together consistently. **One alpha source of truth: the canvas-level alpha.**

### Pitfall 4: Show during fade-in race / hide during fade-out race

**What goes wrong:** User rapid-presses (press, release, press, release, press) within < 200ms. Multiple `:show(0)`/`:hide(0.15)` calls overlap. NSAnimationContext blocks queue up; HUD ends up in a weird half-visible state.

**Why it happens:** `:hide(0.15)` runs an NSAnimationContext block over 150ms; if `:show(0)` is called during that window, NSWindow's `orderFront` and the in-flight animator collide.

**How to avoid:** The `isRecording` re-entrancy guard (line 71 + 268 of `init.lua`) already prevents the rapid-press case at the *recording* layer. As long as `showHUD()` is gated on `not isRecording → onPress proceeds`, the HUD lifecycle is gated too. But for defence-in-depth, you can pre-empt the race by setting alpha directly:

```lua
local function hideHUD()
  if not hudCanvas then return end
  hudCanvas:hide(0.15)  -- normal path; sufficient given isRecording gate
  -- If race shows up: hudCanvas:alpha(0):hide(0)  -- snap to invisible immediately
end
```

The simple `:hide(0.15)` is adequate; document this pitfall but don't over-engineer for v1.

### Pitfall 5: Forgetting `wantsLayer(true)` → CPU pegged when visible

**What goes wrong:** Without `wantsLayer(true)`, `hs.canvas` falls back to the AppKit drawing path, which involves more CPU draw calls per frame. Visible-state CPU usage may spike to 5-10% on multi-monitor setups.

**Why it happens:** `wantsLayer` defaults to `false` (per `hs.canvas` docs); needs explicit opt-in.

**How to avoid:** Always call `:wantsLayer(true)` at canvas creation. Free win — engages Core Animation's GPU compositing, drops CPU usage to ~0%.

### Pitfall 6: macOS Sequoia 15.7.5 hot-reload edge case (Hammerspoon ~6936)

**What goes wrong:** `hs.reload()` after editing `init.lua` may leave the previous canvas instance in a half-torn-down state if the canvas was visible at reload time. Symptom: a ghost lavender pill stuck at the old position.

**Why it happens:** Hammerspoon's reload tears down the Lua state but the underlying NSWindow may not get released until GC — especially if the canvas was mid-animation when reload fired.

**How to avoid:**
1. The defensive `if _G._purplevoice_hud then ... :delete() ... end` block at module load (Pattern 1) catches this — explicit `:delete()` orders out and releases the NSWindow before the new canvas is created.
2. Empirical reload test: load `init.lua`, press hotkey, while HUD visible run `hs.reload()` from console, observe — should see the old pill vanish before the new pill appears, no ghost.

### Pitfall 7: Position arithmetic forgetting global coordinates

**What goes wrong:** Multi-monitor user, focused window on right monitor (`screenFrame = { x = 1920, y = 0, w = 1920, h = 1080 }`); position math uses `(screenFrame.w - HUD_W) / 2` directly without adding `screenFrame.x`. HUD ends up at x = 890 (within left monitor's frame), not at x = 2810 (centered on right monitor).

**Why it happens:** In single-monitor-land, `screenFrame.x = 0` always, so the bug doesn't surface. Surfaces immediately on dual-monitor.

**How to avoid:** Always add `screenFrame.x` (and `screenFrame.y` for vertical math) to the computed coordinate. Pattern 5 in this document does this correctly; the planner must not refactor that math without preserving the offsets.

### Pitfall 8: `hs.styledtext` font key spelling

**What goes wrong:** Setting `font = "System"` (string instead of table) → text renders in the default system font but at the *default* size (~12pt) — too small for a 36px-tall pill.

**Why it happens:** `hs.styledtext.new()` accepts `font` as either a string (font name only, default size) or a table `{ name = "...", size = N }`. The string form is a footgun for size-sensitive uses.

**How to avoid:** Always use the table form: `font = { name = ".AppleSystemUIFont", size = 14 }`. The leading-dot form `.AppleSystemUIFont` is the system font with macOS-default-weight; matches what `hs.alert` uses by default.

### Pitfall 9: `near-cursor` position when cursor is near screen edge

**What goes wrong:** Cursor at top of screen → `near-cursor` puts HUD at `y = mp.y + 30`, which can be off-screen if cursor is in menubar (y ≈ 0 to 25). Worse: cursor at right edge → x = mp.x - 70 may push HUD off the right side.

**Why it happens:** Naive arithmetic doesn't clamp.

**How to avoid:** Clamp the result to fit within the cursor's screen frame:

```lua
elseif name == "near-cursor" then
  local mp = hs.mouse.absolutePosition()
  local cursorScreen = hs.screen.find(mp) or hs.screen.mainScreen()
  local cf = cursorScreen:fullFrame()
  local x = math.max(cf.x + EDGE_MARGIN, math.min(mp.x - HUD_W / 2, cf.x + cf.w - HUD_W - EDGE_MARGIN))
  local y = math.max(cf.y + EDGE_MARGIN, math.min(mp.y + NEAR_CURSOR_OFFSET, cf.y + cf.h - HUD_H - EDGE_MARGIN))
  return { x = x, y = y }
end
```

**Recommendation:** Include the clamp in v1 — it's 5 lines, prevents user confusion when the HUD appears partly off-screen.

### Pitfall 10: Brand consistency lint catches new `voice-cc` references

**What goes wrong:** A maintainer copies an old test pattern that uses `VOICE_CC_HUD_OFF` (legacy naming) → `tests/test_brand_consistency.sh` fails on commit.

**Why it happens:** Phase 2.5 D-09 enforces `PURPLEVOICE_*` namespace for all env vars; the brand-consistency test explicitly grep-checks for `voice-cc` strings.

**How to avoid:** All new env vars MUST use `PURPLEVOICE_` prefix (D-10 already locked). The lint catches this at commit time; pre-commit grep `! grep -rn "VOICE_CC_HUD\|voice-cc-hud" purplevoice-lua/ tests/` would catch it earlier.

### Pitfall 11: Manual-walkthrough validation files missing from `tests/manual/`

**What goes wrong:** Plan adds `tests/manual/test_hud_*.md` files in execution but skips them in the plan's verify clause. Reader cannot verify the manual-walkthrough deliverables exist.

**Why it happens:** Manual walkthroughs aren't auto-aggregated by `tests/run_all.sh` (which only globs `tests/test_*.sh`). Easy to forget.

**How to avoid:** Plan verify clauses should explicitly check `[ -f tests/manual/test_hud_appearance.md ]` etc. Update VALIDATION.md (Nyquist gate) to track manual-walkthrough completion.

### Pitfall 12: `hs.canvas` element coordinate space is canvas-local, not screen-global

**What goes wrong:** Setting the rectangle element's `frame.x = 200` thinking it's the canvas's *screen* position. Element coordinates are *relative to the canvas top-left* — element frame `{x = 0, y = 0, w = 140, h = 36}` fills the entire canvas; `{x = 200, y = 0, ...}` puts the element off the right edge of the 140-wide canvas.

**Why it happens:** The canvas's `frame()` method uses screen-global coordinates; the element's `frame` field uses canvas-local coordinates. Same word, different scope.

**How to avoid:** Element `frame`s for the rectangle and text should always be relative to `(0, 0)` (the canvas's own top-left). Use the canvas's `:topLeft({x, y})` to position the canvas in screen coordinates.

   - What we know: Phase 2 Plan 02-03 ran live walkthroughs with user sign-off (TCC mic deny, Accessibility deny). Phase 2.5 P04 did *not* run live walkthroughs (string-level only).
   - What's unclear: Phase 3.5 has more user-visible interactive surface than 2.5 but less safety-critical than 2 (no permissions involved). The `03.5-CONTEXT.md` Claude's Discretion entry mentions "First-load smoke test approach — manual walkthrough vs an automated screenshot-pixel-check vs visual sanity-check during install. Planner picks based on RESEARCH automation feasibility." — research finds: pixel-check is brittle (Apple changes anti-aliasing across versions), visual sanity-check during install is ad-hoc; the right answer is **structured manual walkthroughs as part of phase verification**.
   - Recommendation: Wave 3 plan executes 3 of the 5 walkthroughs live (test_hud_appearance, test_hud_disable, test_hud_focus) and signs them off in the SUMMARY.md. The other 2 (test_hud_idle_cpu, test_hud_screen_capture) are documentation-only — they document the expected behaviour and the limitation, no live demonstration required.

2. **Should the SECURITY.md HUD caveat live in §"How to Verify These Claims" or in a new H2 section §"HUD Visibility in Screen Recordings"?**
   - What we know: SECURITY.md has 18 H2 sections (per Phase 2.7 closure); adding a 19th requires updating the framing lint's required-sections list (`tests/test_security_md_framing.sh`).
   - What's unclear: pure stylistic call. New section = more discoverable; nested in §"How to Verify" = less disruptive to existing structure.
   - Recommendation: extend §"How to Verify These Claims" with a sub-paragraph that links to the README `## HUD Configuration` subsection. Avoids growing SECURITY.md by an H2 section AND avoids touching the framing-lint required-sections list.

3. **Does the planner want to add `hudCanvas:setBehavior(...)` updates on space changes (e.g., listening for `hs.spaces.watcher`) to handle the "user switches Spaces mid-press" edge case?**
   - What we know: `canJoinAllSpaces` makes the canvas appear in all Spaces simultaneously. Verified via Hammerspoon docs.
   - What's unclear: with `canJoinAllSpaces` set, no watcher is needed — the canvas is already pinned across Spaces.
   - Recommendation: Don't add a watcher. `canJoinAllSpaces` is sufficient; an `hs.spaces.watcher` would add complexity for no behavioural difference.
