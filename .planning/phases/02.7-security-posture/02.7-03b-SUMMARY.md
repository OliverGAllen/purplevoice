## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Plan-prose-vs-verify inconsistency] FedRAMP table cell vocabulary**

- **Found during:** Task 3b-1 (FedRAMP section)
- **Issue:** Plan example used `| NIST 800-53 Low-baseline alignment | Compatible (see ...) |` — the must_haves require Met / Partial / Not Pursued / N/A only. While the framing lint only blocks `Compliant / Certified / Compliance` in table cells, the must_haves vocabulary discipline forbids "Compatible" as a status value.
- **Fix:** Changed to `| NIST 800-53 Low-baseline alignment | Met (see ...) |`.
- **Rationale:** Met is the honest framing — the NIST 800-53 Low-baseline mapping in Plan 03a substantiates this status; the value is Met, the verbose rationale clarifies "see ... mapping".
- **Files modified:** SECURITY.md
- **Commit:** `eb1ec75`

**2. [Rule 1 — Plan-prose-vs-verify inconsistency] HIPAA section "HIPAA-compliant entity" / "HIPAA-compliant systems"**

- **Found during:** Task 3b-2 (HIPAA section)
- **Issue:** Plan template contained the phrase "PurpleVoice is **a tool** — it cannot itself be a HIPAA-compliant entity" and "PurpleVoice ... makes it **a supporting safeguard** for HIPAA-compliant systems handling voice notes / dictation workflows". Both contain `\bcompliant\b` outside any qualified context, which the framing lint rejects (`PASS [test_security_md_framing.sh]: ... clean` requires zero unqualified hits).
- **Fix:** Rephrased to "PurpleVoice is **a tool**, not a HIPAA-regulated entity" and "supporting safeguard for HIPAA-aligned systems".
- **Rationale:** "HIPAA-regulated entity" preserves the meaning ("the regulated party is the Covered Entity, not the tool") without the banned phrase. "HIPAA-aligned" is the intended framing throughout this document anyway. Same root cause / same fix pattern as Plan 02.7-00's framing-explainer rephrase and Plan 02.7-03a's SC-12(2) prose rephrase: when a plan's prose template contains a literal banned string that the same plan's verify regex forbids, the prose must be rephrased to a banned-phrase-free form.
- **Files modified:** SECURITY.md
- **Commit:** `63a978b`

**3. [Rule 1 — Plan-prose-vs-verify inconsistency] SOC 2 section "A tool cannot be SOC 2 compliant"**

- **Found during:** Task 3b-2 (SOC 2 section)
- **Issue:** Plan template contained the phrase "**A tool cannot be SOC 2 compliant.**" as the defensive Pitfall 11 disclaimer line. Same `\bcompliant\b` issue as deviation #2. The framing lint allows "not compliant" / "non-compliant" as qualified contexts, but "SOC 2 compliant" doesn't match the qualifier regex. Then I tried `"compliant"` (quoted form, which the framing lint allows) but the plan's stricter verify clause `! grep -nE '\bcompliant\b' SECURITY.md` rejects ANY occurrence regardless of quoting. Two-step fix.
- **Fix:** Rephrased to "**A tool does not hold a SOC 2 report — only the organisation operating it can.**". Same defensive Pitfall 11 framing without the banned phrase.
- **Rationale:** The intent — defensive disclaimer that a tool cannot self-attest SOC 2 — is preserved. SOC 2 is a *report* issued to organisations after audit; a tool can be in the system boundary but cannot itself hold the report. This phrasing is arguably more precise than the plan template anyway.
- **Files modified:** SECURITY.md
- **Commit:** `63a978b`

> **If a plan's prose template contains a literal string that the same plan's verify regex (or the framing lint installed by an earlier plan) forbids, the prose must be rephrased to a banned-phrase-free form — even when the meaning is the same.**

This is the third plan in the phase (after 02.7-00 and 02.7-03a) where this deviation has been needed; it should now be a checker-pre-flight item rather than an in-flight discovery. Previously seen:
