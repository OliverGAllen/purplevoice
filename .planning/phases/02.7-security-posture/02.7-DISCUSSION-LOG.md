## SBOM Tooling & Format

### Q1: What tool generates the SBOM?

| Option | Description | Selected |
|--------|-------------|----------|
| Syft (Recommended) | Anchore's Syft — ecosystem-standard, free, scans filesystem + Homebrew + binaries, outputs SPDX or CycloneDX. Auditor-recognised. Optional dep in setup.sh. | ✓ |
| Hand-authored markdown table | Curated docs/SBOM.md table. Lowest friction; less auditor-credible. | |
| CycloneDX CLI | Official OWASP tooling. Stronger security-attestation features; less filesystem-scanner-friendly than Syft. | |
| Hybrid: Syft + markdown summary | Syft generates SBOM.spdx.json; hand-curated SBOM.md provides human summary. Two artifacts, one source of truth. | |

**User's choice:** Syft (Recommended)

### Q2: Which SBOM format gets committed to the repo?

| Option | Description | Selected |
|--------|-------------|----------|
| SPDX-JSON (Recommended) | ISO/IEC 5962:2021, US gov / NTIA-aligned. The format federal procurement officers recognise. File: SBOM.spdx.json. | ✓ |
| CycloneDX-JSON | OWASP standard, security-tooling-friendly, supports vulnerability attestations. Less recognised by gov procurement. | |
| Both formats | Maximum compatibility; two files to keep in sync. | |
| Human-readable markdown only | Lightest; auditors can't consume programmatically. | |

**User's choice:** SPDX-JSON (Recommended)

### Q3: What's the SBOM scope — how deep do we go?

| Option | Description | Selected |
|--------|-------------|----------|
| Direct runtime deps + provenance (Recommended) | sox, whisper-cli, models, Hammerspoon, scripts. Each with version, source URL, SHA256, license. Stops at first-level deps. | |
| Direct + transitive build deps | Above PLUS whisper.cpp's compile deps, sox's audio libs. Larger; Syft auto-discovers. | |
| Direct + system context | Direct deps PLUS macOS version, hardware platform, Xcode CLT version. "What is the trusted compute base." | |
| Full: direct + transitive + system | Maximum scope. Strongest claim for an auditor. Highest maintenance burden. | ✓ |

**User's choice:** Full: direct + transitive + system

**Notes:** This is the only deviation from the recommended option. The user explicitly chose maximum scope to maximise auditor credibility — this is a deliberate upgrade with known maintenance cost. The planner should budget Syft auto-discovery time for transitive deps and system-context capture (macOS version, hardware variant, Xcode CLT version).

### Q4: When does the SBOM get regenerated?

| Option | Description | Selected |
|--------|-------------|----------|
| setup.sh regenerates if Syft present (Recommended) | setup.sh runs syft scan on every run if Syft installed; prints skip notice if not. SBOM committed; drift visible in git diff. | ✓ |
| Manual regen on dep change | Document the regenerate command. Risk: SBOM goes stale silently. | |
| Release-tag only | SBOM regenerated when tagging a release. Tied to git tags. | |
| GitHub Actions auto-regen | CI runs Syft on every push, fails if SBOM drifted. Strongest signal; adds CI infra. | |

**User's choice:** setup.sh regenerates if Syft present (Recommended)

**Notes:** Three of four recommendations accepted; SBOM scope upgraded to Full. User opted to skip "More questions about SBOM" and move to next area.
