## Phase Boundary

**What this phase delivers:** `SECURITY.md` as the authoritative security document for PurpleVoice — substantiating the institutional / government / healthcare / legal / finance / journalist / air-gapped audience claims with a threat model, runnable verification mechanisms, and an honest gap analysis against seven security frameworks.

**Phase 2.7 IS:**
- An authoritative `SECURITY.md` published in repo root
- A directory of runnable verification scripts (`tests/security/`) that any reader can execute on their own machine to confirm each PurpleVoice claim
- A committed SBOM (`SBOM.spdx.json`) covering direct + transitive + system-context dependencies
- A gap-analysis section mapping PurpleVoice posture against NIST SP 800-53 (deep) + 6 other frameworks (framed)
- An air-gap operability path: `PURPLEVOICE_OFFLINE=1` mode in `setup.sh` that skips downloads + accepts a sideloaded model file

**Phase 2.7 is NOT:**
- Pursuit of formal certifications (FedRAMP authorisation, FIPS 140-3 validation, Common Criteria certification, SOC 2 Type II audit, ISO 27001 certification) — these are 6–18 month, $50k+ commitments and disproportionate for a free open-source tool. The framing is "compatible with applicable [framework] obligations" — never "compliant".
- Code signing + notarisation infrastructure — that work belongs to Phase 3 (Distribution) when the installer artifact actually ships. Phase 2.7 documents expectations but does not stand up Apple Developer ID signing.
- A vulnerability disclosure programme — single-developer tool; revisit if a community emerges.
- A user-facing UI surface — pure docs + verification scripts.
- GitHub Actions CI integration — release-gate verification (manual) is the chosen cadence; CI infrastructure is Phase 3 territory.

**Requirements covered (proposed in ROADMAP, formalised during planning):** SEC-01, SEC-02, SEC-03, SEC-04 (documented-only this phase), SEC-05 (documented gaps + air-gap path), SEC-06 (air-gap operability sideload).

**Success criteria (from ROADMAP §"Phase 2.7"):**
1. `SECURITY.md` published in repo root with all sections per SEC-01.
2. A reader (journalist, sysadmin, government IT auditor) can independently verify the zero-egress claim by running `tests/security/run_all.sh` and reading the methodology in `SECURITY.md`.
3. `SBOM.spdx.json` exists in repo root, lists all runtime dependencies (direct + transitive + system context) with versions, source URLs, SHA256s, licenses.
4. Code signing + notarisation status is documented honestly as "deferred to Phase 3 — applies when installer artifact ships".
5. PROJECT.md positioning paragraph (already shipped in Phase 2.5 BRD-01) is verified against the SECURITY.md substantiation.

<decisions>
## Implementation Decisions

### Verification Artifacts

- **D-01:** **Deliverable shape = `SECURITY.md` + runnable scripts in `tests/security/`.** Authoritative narrative document + automated verification scripts a reader executes on their own machine. No captured reference outputs (pcap files, screenshots, SBOM diffs) committed to the repo — committed evidence goes stale fast (next macOS update, new dep version) and a stale pcap is worse than no pcap. The scripts ARE the auditable surface; readers reproduce from clean state.

- **D-02:** **No reference outputs in git.** Verification is reproducer-driven, not evidence-driven. The repo ships methodology + scripts; readers generate evidence themselves on their own machines. Exception: a single Syft-generated `SBOM.spdx.json` IS committed (it's the artifact, not evidence of the artifact).

- **D-03:** **Verification cadence = release-gate.** Scripts run before tagging a release; failures block release. No per-commit GitHub Actions CI in this phase (Phase 3 territory). Manual ad-hoc runs by the developer or external auditors are also supported. Keeps CI infra cost zero while preserving the verifiability claim.

- **D-04:** **Script location = `tests/security/`** with a `tests/security/run_all.sh` aggregator that mirrors the existing `tests/run_all.sh` pattern. New scripts: `tests/security/verify_egress.sh`, `tests/security/verify_sbom.sh`, `tests/security/verify_air_gap.sh` (and any others surfaced during planning). Each script exits 0 on pass, non-zero with a descriptive failure message on fail.

### Egress Proof

- **D-05:** **Primary egress proof tool = `pf` rules + `tcpdump` capture script.** Built-in macOS, free, scriptable. `tests/security/verify_egress.sh` configures a `pf` anchor that blocks all outbound for the PurpleVoice process group, runs `tcpdump` on the loopback + Wi-Fi + Ethernet interfaces during a synthesised recording session, then parses the capture for any leaked packets attributable to the PurpleVoice process tree. Requires `sudo`. README + script preamble document the sudo requirement prominently. Strongest evidence available without commercial dependencies.

- **D-06:** **Exit criterion = process-tree network silence.** The "no egress" assertion is tightly scoped to `purplevoice-record` + its child PIDs (`sox`, `whisper-cli`) + Hammerspoon's Lua-spawned subprocesses, NOT system-wide. Specifically: zero open TCP/UDP sockets, zero DNS resolutions, zero `connect()` syscalls within the process tree during a recording window. System-wide capture is included for transparency but the assertion is on the PurpleVoice process tree only — avoids mDNS / system-telemetry / unrelated-app noise muddying the claim.

- **D-07:** **Egress test scope = runtime only.** The privacy claim is about voice content during recording. `setup.sh`'s downloads (HuggingFace `ggerganov/whisper.cpp`, Homebrew `brew.sh`, GitHub for Hammerspoon cask) are explicit, one-time, documented in `SECURITY.md` by hostname + URL. The runtime egress test verifies the recording-time silence. The `setup.sh` contacts are framed in narrative, not script-verified, to keep the test surface small.

- **D-08:** **Air-gap operability via `PURPLEVOICE_OFFLINE=1` mode in `setup.sh`.** When the env var is set, `setup.sh` skips the model download step + the Silero VAD download step + the Hammerspoon brew install, and instead accepts a sideloaded model file at a documented path (e.g., `~/.local/share/purplevoice/models/ggml-small.en.bin` placed by the operator beforehand). README + SECURITY.md document the air-gap install procedure. This substantiates the air-gapped operator audience claim concretely.

### SBOM (Software Bill of Materials)

- **D-09:** **SBOM tool = Syft (Anchore).** Free, ecosystem-standard, federal-recognised. Scans filesystem + Homebrew + binaries + macOS system context. Output format SPDX-JSON. Adds `syft` as an optional dep checked by `setup.sh` (if not installed, `setup.sh` prints "Syft not found, SBOM regen skipped" and continues — the committed `SBOM.spdx.json` from a prior regen still holds).

- **D-10:** **SBOM format = SPDX-JSON, committed as `SBOM.spdx.json` in repo root.** ISO/IEC 5962:2021, US NTIA-aligned, the format federal procurement officers recognise. Single committed format keeps maintenance simple. CycloneDX consumers can convert via Syft's `-o cyclonedx-json` flag at read-time without committing a second artifact.

- **D-11:** **SBOM scope = full (direct + transitive + system context).** This is an upgrade from the recommended baseline. Direct: `sox`, `whisper-cli`, `ggml-small.en.bin`, `ggml-silero-v6.2.0.bin`, `Hammerspoon.app`, `purplevoice-record`, `purplevoice-lua/init.lua`. Transitive: Syft auto-discovers compile-time deps of whisper.cpp (ggml internals), sox's audio libs (libsndfile, libvorbis, etc.), Hammerspoon's bundled Lua + LuaSocket. System context: macOS version, hardware platform (Apple Silicon variant — M1 / M2 / M3 / M4), Xcode CLT version, brew version. **Maintenance burden:** higher than direct-only; auditor credibility is correspondingly higher. Government-audit framing: "what is the trusted compute base".

- **D-12:** **SBOM regeneration cadence = `setup.sh` regenerates if Syft present.** Idempotent: if `syft` is on PATH, `setup.sh` runs `syft scan` against the repo + relevant filesystem locations and writes `SBOM.spdx.json` (deterministic output). If Syft is absent, prints a one-line skip notice and continues. SBOM is committed; drift between regenerations is visible in `git diff` and reviewed manually before release.

### Government Framework Gap Analysis

- **D-13:** **Primary reader = Technical IT auditor.** Sysadmin / security engineer at a government / healthcare / finance org evaluating PurpleVoice for use. This audience reads the threat model, runs the verification scripts, scrutinises the control mappings, and demands honest gap analysis. Document tone is technical + transparent. This audience covers other named audiences by default — a journalist or end user can skim the TL;DR; a procurement officer can read the gap-analysis tables; the technical auditor reads everything.

- **D-14:** **Depth model = NIST SP 800-53 Rev 5 / Low-baseline primary; six other frameworks framed only.** NIST 800-53 gets per-control mapping in depth (the US fed reference; most reusable across the broadened audience). The other six frameworks (FIPS 140-3, FedRAMP-tailored, Common Criteria, HIPAA Security Rule §164.312, SOC 2 Type II Trust Services Criteria, ISO/IEC 27001 Annex A) each get a 1-2 paragraph "where PurpleVoice stands" framing without per-control mapping. Realistic for the phase budget while serving the international audience (ISO 27001) + healthcare audience (HIPAA) + finance audience (SOC 2) + US fed audience (NIST + FIPS + FedRAMP + Common Criteria).

- **D-15:** **Format = hybrid (prose + summary table).** Each framework gets a narrative section (what the framework is, where PurpleVoice stands, gaps + rationale) AND a summary table mapping PurpleVoice claims → framework control IDs. Readable for humans + scannable for compliance officers. NIST 800-53's table is the most detailed; the other six are summary-only.

- **D-16:** **Frameworks covered = seven total** (expanded per user direction "open to any country"):
  1. **NIST SP 800-53 Rev 5** (Low-baseline) — deep per-control mapping
  2. **FIPS 140-3** — framed; PurpleVoice does not perform cryptographic operations on user data, so most clauses are N/A; framing is "compatible with FIPS-validated cryptographic modules where the underlying macOS crypto APIs are FIPS-validated"
  3. **FedRAMP-tailored** — framed; FedRAMP is an authorisation programme, not a control catalogue (it inherits NIST 800-53); framing is "achievable IF a sponsoring agency pursues authorisation; not unilaterally pursuable"
  4. **Common Criteria** — framed; product certification, not framework mapping; framing is "out of scope for v1; PurpleVoice's small TCB and open-source posture would simplify a future ST-based evaluation"
  5. **HIPAA Security Rule §164.312 (Technical Safeguards)** — framed; PurpleVoice does not store or transmit PHI by design, so most safeguards (access control, audit, integrity, transmission security) are N/A or trivially Met; framing is "compatible with §164.312 obligations for the in-scope technical safeguards"
  6. **SOC 2 Type II Trust Services Criteria** — framed; SOC 2 reports are issued to organisations, not tools; framing is "compatible with applicable SOC 2 TSC for in-scope code (Security, Confidentiality, Privacy partially)" — never "SOC 2 compliant"
  7. **ISO/IEC 27001:2022 Annex A controls** — framed; ISO 27001 certification is to ORGANISATIONS, not tools; framing is "compatible with applicable Annex A technical controls (A.8 Access Control, A.13 Communications Security, A.14 System Acquisition Development Maintenance) for in-scope code" — useful for international / EU / UK / AU institutional buyers

- **D-17:** **"Compatible with" framing is mandatory and enforced consistently.** No framework section uses the word "compliant" without an explicit qualifier. Over-claiming would damage the entire "auditable + verifiable" trust narrative that PurpleVoice's positioning depends on. The plan-checker / planner should verify this language constraint during planning.

### Claude's Discretion

- **Threat model framework choice** — ROADMAP mentions "STRIDE-style threat enumeration"; Claude picks STRIDE primary during planning research; optional LINDDUN privacy-threat overlay if research suggests it adds value for the privacy-first positioning.
- **Specific NIST 800-53 control IDs** mapped — Researcher / planner picks the relevant subset from AC, AU, IA, SC, SI families; not every control applies to a single-binary local tool.
- **Code signing scope this phase** — Phase 2.7 documents Apple Developer ID + notarisation expectations + cost ($99/yr) + Phase 3 deferral rationale. No infrastructure work this phase. Planner has discretion to add a stub `tests/security/verify_signing.sh` that asserts "no signed artifacts yet — applies in Phase 3".
- **Reproducible build (SEC-05) handling** — Document goal + current gaps in SECURITY.md; full byte-identical reproducibility for whisper.cpp Metal-compiled artifacts is hard (toolchain-version-sensitive). Planner has discretion to scope a pragmatic "best-effort with these caveats" verification.
- **Plan boundaries** — ROADMAP suggests 4 plans (threat model, verification mechanisms, gap analysis, docs finalisation). With 7 frameworks (vs the 4 in ROADMAP) the gap-analysis plan may be heavier; planner has discretion to split into 5 plans or keep 4 with appropriate task-level breakdown.
- **Exact wording of "compatible with" disclaimers** — Planner / researcher refines per framework; the constraint is consistency, not specific phrasing.
- **TL;DR / one-pager design** — SECURITY.md should open with a 1-page TL;DR for non-technical readers; format/structure is Claude's discretion.

<specifics>
## Specific Ideas

- **"Auditable + verifiable" framing is the load-bearing differentiator.** PROJECT.md uses this language deliberately; SECURITY.md must reinforce it. Every claim in the document is either (a) substantiated by a runnable script in `tests/security/`, or (b) explicitly framed as "compatible with [framework]" / "out of scope, here's why" / "deferred to Phase X". No unsubstantiated claims.

- **The "compatible with" language enforcement is critical.** With seven frameworks (especially SOC 2 + ISO 27001 which are organisational standards), the temptation to slip into "compliant" or "certified" claims is real. The plan-checker and planner must enforce this consistently. A misframed claim damages the entire trust narrative.

- **Government audience reads the gap analysis backwards.** A US federal IT auditor scans the NIST 800-53 mapping table first, the threat model second, the verification scripts third. An EU institutional buyer reads the ISO 27001 framing first. SECURITY.md's structure should let each audience find their entry point fast.

- **The pf+tcpdump approach is the strongest evidence available for free.** Little Snitch would give visual evidence but is paywalled and contradicts the open-source ethos. lsof+nettop is faster but proves "nothing connected" not "nothing transmitted". The chosen approach trades sudo friction for the strongest credible claim.

- **NIST 800-53 Low-baseline is the realistic target.** Moderate and High baselines invoke heavy controls (continuous monitoring, audit-log retention, incident response procedures) that don't apply to a single-binary local tool. Claiming Low-baseline alignment is honest; claiming Moderate would over-reach.

- **The air-gap path matters more than it seems.** A documented `PURPLEVOICE_OFFLINE=1` mode is a single-script-edit cost but a load-bearing claim for the air-gapped operator audience (defence, intelligence, classified networks). Without it, the audience claim is rhetoric.

- **SBOM full scope (D-11) is a deliberate upgrade.** The user chose direct + transitive + system over the recommended direct-only because government audits frame it as "what is the trusted compute base". The maintenance cost is real (Syft must run on every dep change to catch transitives) but the auditor credibility scales accordingly.

- **The user explicitly broadened to "any country" (international reach)** — this is why ISO/IEC 27001 is in scope. The SOC 2 + ISO additions cover non-US institutional buyers; the framing must work for European GDPR-conscious orgs (privacy-by-design language), Five-Eyes intel community (NIST is recognised but ISO is also expected), and Asia-Pacific buyers (ISO + Common Criteria common).

</specifics>

Achieving full reproducibility would require pinning a specific toolchain in setup.sh — disproportionate for v1. Phase 2.7 documents the goal + current gaps + a "best-effort" verification (re-run setup, diff installed file SHA256s — known to fail across toolchain versions). Full reproducibility is a v2 candidate.

- **Vulnerability disclosure programme** — `SECURITY.md` will include a "How to report a vulnerability" stub (email contact to oliver@olivergallen.com) but no CVE-issuance authority, bounty programme, or PGP key infrastructure. Single-developer tool; revisit if a community emerges (Phase 3+ or v2).

- **Per-control mapping for FIPS 140-3 / FedRAMP / Common Criteria / HIPAA / SOC 2 / ISO 27001** — Only NIST 800-53 gets per-control depth (D-14). The other six are framed at the section level. Per-control mapping for any of the six would be its own phase if a sponsoring institution requests it.

- **LINDDUN privacy threat modelling overlay** — STRIDE primary per ROADMAP. LINDDUN (privacy-specific threat categories) is a Claude's-discretion add during planning if research suggests it would strengthen the privacy-first positioning. Not blocking.

- **Threat model formal review by external party** — A formal STRIDE workshop with security experts is heavier than this phase budget. SECURITY.md's threat model is single-developer-authored + open for community review via GitHub issues. Formal review is a v2 candidate.

- **PASTA / OCTAVE / other threat-modelling methodologies** — STRIDE is the chosen methodology (per ROADMAP). Other methods are out of scope.
