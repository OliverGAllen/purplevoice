## SBOM Scope-Gap Finding (Routed to Human Review)

This is the single most material finding in the verification — repeated here for visibility.

**Claim under examination:** SECURITY.md §SBOM (lines 209-219) describes the SBOM scope as "Direct dependencies: sox, the local transcription binary, ggml-small.en.bin, ggml-silero-v6.2.0.bin, Hammerspoon.app, purplevoice-record, purplevoice-lua/init.lua. Transitive dependencies: ... Auto-discovered by Syft." (D-11 full scope from CONTEXT.md).

**Actual artifact:** `SBOM.spdx.json` `packages` array contains 1 entry: `{name: "PurpleVoice", versionInfo: <git-short-sha>}`. 4 system-context Annotation blocks (macOS-version, hardware-platform, xcode-clt-version, brew-version). No separate package entries for `sox` / transcription binary / Hammerspoon / GGML model / transitives.

**Cause:** Plan 02 chose Syft scope = `dir:$REPO_ROOT` (per Pitfall 2 — naive `dir:$HOME` would leak unrelated brew formulae across Oliver's machine into the SBOM). The repo doesn't *contain* the binaries or models — they live in `/opt/homebrew/Cellar/{sox,whisper-cpp}` and `/Applications/Hammerspoon.app` and `~/.local/share/purplevoice/models/`. Syft scoped to repo only sees the repo's text files, all of which roll up into the single `PurpleVoice` package descriptor.

**Plan SUMMARY transparency:** Plan 02 (`02.7-02-SUMMARY.md`) explicitly states: "1 package (PurpleVoice itself, version=git-short-sha) + 4 SPDX 2.3 Annotation blocks (macOS 15.7.5 24G624 / arm64 Apple Silicon / Xcode CLT 26.2.0.0.1.1764812424 / brew 5.1.8)." The SUMMARY is internally consistent.

**Pitfall-2 RESEARCH guidance:** "The planner picks a single primary scan invocation for setup.sh Step 8; the multi-scope invocation is documented as a release-only 'comprehensive SBOM regen' target for thoroughness." The release-only multi-scope variant is NOT implemented.

**Auditor impact:** A government IT auditor reading SECURITY.md §SBOM expects to find sox / Hammerspoon / GGML deps as separate enumerated package entries in the JSON, then jq-filter / Syft-convert to extract them. Opening the JSON and finding 1 package will produce a "wait, where's the rest?" reaction. The system-context annotations help (they tell the auditor what the host OS + brew + Xcode CLT context is) but don't substitute for direct/transitive dep enumeration.

**Mitigation options for v1 release:**

1. **Accept the gap** (current state). Add a one-paragraph "What this SBOM contains and what it doesn't" disclaimer to SECURITY.md §SBOM explaining the repo-only scope choice, framing system-context annotations as the OS+toolchain trust-baseline, and pointing to brew + Hammerspoon's own SBOMs for transitive depth. This is honest engineering — the trade-off is documented and the auditor knows to look upstream.
2. **Extend Step 8 to multi-scope scan** before v1 release. Add `syft scan dir:/opt/homebrew/Cellar/sox dir:/opt/homebrew/Cellar/whisper-cpp dir:/Applications/Hammerspoon.app -o spdx-json --merge` (or equivalent) at release-gate time only, producing an enriched SBOM. Cost: 30-60s of Syft scan time at release; benefits: prose-vs-artifact alignment.
3. **Compromise:** Ship v1 with current scope; add a multi-scope `verify_sbom_full.sh` opt-in script that release-time-only auditors can run; commit the enriched SBOM as a release artifact in v1.0 tag.

**Verifier recommendation:** Option 1 (one-paragraph disclaimer) is the lowest-friction path that preserves goal achievement. Option 2 is the strongest signal but adds a release-time dependency on multi-scope Syft + post-process. Final call belongs to Oliver.

This finding is routed to `human_verification` item #2 in frontmatter rather than as a `gap` because all 5 success criteria + all 6 SEC requirements ARE met as written (SBOM file exists; verify_sbom.sh validates the artifact as it actually exists; SECURITY.md §SBOM scope claim is aspirational rather than literally false — Syft auto-discovers what's in scope). The decision is editorial, not a regression.
