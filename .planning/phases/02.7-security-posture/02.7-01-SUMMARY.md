## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Brand-consistency test classified the framing lint as a brand violation**

- **Found during:** Task 1-2 verify clause (final functional-suite run)
- **Issue:** After implementing the framing lint, `bash tests/run_all.sh` reported `test_brand_consistency.sh: FAIL` with diagnostic `voice-cc strings found in non-exempt source files: ./tests/test_security_md_framing.sh`. The new lint legitimately contains the literal string `voice-cc` because (a) Check 5's grep pattern `grep -q "voice-cc"` is load-bearing — it must contain the exact string it is detecting; (b) the failure-message text and the section-comment legitimately reference the brand-consistency invariant being enforced.
- **Fix:** Added `tests/test_security_md_framing.sh` to `tests/test_brand_consistency.sh`'s exemption list, mirroring the existing `tests/test_brand_consistency.sh` self-exemption pattern. This is the same root cause / same fix used when the brand-consistency test was originally introduced (it had to exempt itself for the same reason).
- **Files modified:** `tests/test_brand_consistency.sh` (+4 lines: 1 comment line + 1 grep -v line + 2 surrounding edits to keep the comment table aligned)
- **Commit:** `0a0a58e` (Task 1-2; bundled with the framing lint commit since the exemption is a direct consequence of introducing the lint)
- **Why this is Rule 1 not Rule 4:** This is bug-class — the brand-consistency test as written is incorrectly classifying a legitimate brand-policing test as a brand violation. No architectural change; the exemption mechanism already exists; this is a one-line addition to a list that the test was designed to be extended.

**2. [Rule 1 - Bug] Plan content references whisper-cli; Pitfall 6 discipline says SECURITY.md prose should use transcription-binary framing**

- **Found during:** Task 1-1 (drafting STRIDE / LINDDUN matrices)
- **Issue:** The plan's `<interfaces>` block contained literal `whisper-cli` strings inside the STRIDE Information Disclosure (e) row's mitigation cell ("3-layer evidence chain (lsof + nettop + pf+tcpdump) on the PurpleVoice process tree (`purplevoice-record` + `sox` + `whisper-cli` children)") and in the §Out-of-Scope brew-bottle bullet. Pitfall 6 / Pattern 2 corollary discipline limits literal `whisper-cli` strings to the boundary file `purplevoice-record` so that v1.1 warm-server swap (or any future engine swap) is a one-file-change event. SECURITY.md is a long-lived audit document; baking the current binary name into prose would force future doc edits whenever the binary changes.
- **Fix:** Used "transcription binary" / "transcription children" / "sox / transcription binary" framing in §Scope process lifecycle, §Trust Boundaries, STRIDE Information Disclosure (e), and §Out-of-Scope brew bullet. The semantic precision is preserved (it's still the v1 whisper.cpp build today); the framing is engine-agnostic.
- **Files modified:** `SECURITY.md` (deviation absorbed into Task 1-1 commit)
- **Commit:** `eaac716` (Task 1-1)
- **Why this is Rule 1 not Rule 4:** The framing constraint is documented (Pitfall 6 / Pattern 2 corollary); honouring it is correctness, not architecture. No structural change to the threat model; the matrix still names the same threat with the same mitigation.
