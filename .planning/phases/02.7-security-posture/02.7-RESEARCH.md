### Pattern A: Aggregator-mirror

The `tests/security/run_all.sh` is a **byte-for-byte structural copy** of `tests/run_all.sh` (39 lines, iterates `tests/security/test_*.sh` (rename to `verify_*.sh`), captures pass/fail, exits 0 only if all pass). Reuses the same `set -uo pipefail` (no `-e` so a failure doesn't abort the suite). Logs to `/tmp/purplevoice-security-test.log` (parallel to existing `/tmp/purplevoice-test.log`).

```bash
# tests/security/run_all.sh — exact mirror of tests/run_all.sh
#!/usr/bin/env bash
set -uo pipefail
cd "$(dirname "$0")/../.."   # repo root (one extra ../ for the security/ subdir)

PASS=0; FAIL=0; FAILED=()
for f in tests/security/verify_*.sh; do
  [ -f "$f" ] || continue
  printf "  [security] %-40s " "$(basename "$f")"
  if bash "$f" >/tmp/purplevoice-security-test.log 2>&1; then
    echo "PASS"; PASS=$((PASS + 1))
  else
    echo "FAIL"; FAIL=$((FAIL + 1)); FAILED+=("$f")
  fi
done
echo ""; echo "Results: $PASS passed, $FAIL failed"
if [ "$FAIL" -gt 0 ]; then
  echo "Failed tests:"; printf "  - %s\n" "${FAILED[@]}"; exit 1
fi
exit 0
```

### Pattern B: Helper library

`tests/security/lib/process_tree.sh` (mirroring `tests/lib/sample_audio.sh`) exposes:

```bash
# purplevoice_pid_tree
# Echoes whitespace-separated PIDs of any active purplevoice process tree:
#   - bash glue (purplevoice-record), if running
#   - sox child(ren), if running
#   - whisper-cli child(ren), if running
# Hammerspoon's main process is intentionally NOT included (it's a long-running
# app the user already trusts; PurpleVoice doesn't introduce new behaviour
# there beyond what init.lua does — D-06 scoping decision).
purplevoice_pid_tree() {
  local pids
  pids=$(pgrep -f 'purplevoice-record' 2>/dev/null || true)
  for parent in $pids; do
    pids="$pids $(pgrep -P "$parent" 2>/dev/null || true)"
  done
  echo "$pids" | tr ' ' '\n' | grep -v '^$' | sort -u | tr '\n' ' '
}

# synthesise_recording DURATION
# Triggers a controlled recording for the egress test to capture against.
# Uses PURPLEVOICE_TEST_SKIP_SOX=1 so no actual mic capture happens (the test
# is about NETWORK egress, not audio); the bash glue still spawns whisper-cli
# against a pre-staged WAV. Caller pre-stages a silence WAV at /tmp/purplevoice/recording.wav.
synthesise_recording() {
  local dur="${1:-1.0}"
  source "$(dirname "$0")/../lib/sample_audio.sh"
  silence_wav /tmp/purplevoice/recording.wav "$dur"
  PURPLEVOICE_TEST_SKIP_SOX=1 bash "$REPO_ROOT/purplevoice-record" >/dev/null 2>&1 &
  echo $!  # PID of the bash glue (caller passes to purplevoice_pid_tree wait)
}
```

### Pattern C: Defence-in-depth evidence chain (KEY PATTERN — addresses Pitfall 1)

`verify_egress.sh` does NOT rely on pf alone. Three layers of evidence, any one of which surfacing a leak fails the test:

1. **`lsof -i -p <pid>`** — for each PID in the process tree, snapshot at three sample points during the recording window. **Expected: empty result.** No sudo required.
2. **`nettop -x -P -p <pid> -L 1`** — single-snapshot per-PID network flows. **Expected: zero flow rows for the process tree.** No sudo required.
3. **`pf` anchor + `tcpdump -i any -nn`** — block-all anchor scoped to the macOS UID running the test, captured via tcpdump. **Expected: zero packets attributable to the process tree on `pflog0`.** Requires sudo.

The script exits non-zero if **any** layer surfaces a leak. The script exits 0 only if **all three** layers report silence (or layer 3 is gracefully skipped due to sudo unavailability — see Pitfall 1 sudo handling).

### Anti-Patterns to Avoid

- **Committing pcap files / tcpdump output / SBOM diffs to git.** D-02 forbids this; stale evidence is worse than no evidence. Scripts ARE the auditable surface.
- **Claiming "compliant with" any framework.** D-17 mandates "compatible with" framing. The plan-checker MUST grep for `\bcompliant\b` outside explicit qualifier contexts.
- **Auto-editing `~/.hammerspoon/init.lua`** in air-gap mode (or any mode). Anti-Pattern 4 from PITFALLS.md applies — print instructions, never edit.
- **Relying solely on pf for the egress proof.** Pitfall 1 (Sequoia regression) makes this fragile. The 3-layer evidence chain (lsof → nettop → pf+tcpdump) is mandatory.
- **Running Syft against `~/`** or any path outside the repo + explicit Homebrew Cellar paths. Privacy regression — would leak unrelated installed packages into SBOM.
- **Treating Hammerspoon's main process as in-scope for D-06.** It's a long-running pre-existing app the user trusts; introducing it would expand the assertion to system-wide and dilute the claim.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SBOM generation | Custom script that walks `brew list` + `otool -L` | Syft 1.43.0 (`brew install syft`) | Syft handles direct + transitive + format conversion + checksum + license detection across 30+ ecosystems. Custom = stale within a release. |
| SPDX serialization | Hand-rolled JSON | Syft `-o spdx-json` | SPDX 2.3 schema is large; custom serializers drift from spec. |
| Process-tree enumeration | Walk `/proc` (Linux convention — macOS doesn't have it) | `pgrep -P` + `ps -ef` chains | macOS-native, no portability fiction. |
| Network flow snapshot | Parse raw `/dev/bpf*` | `lsof -i -p PID` + `nettop -x -P -p PID -L 1` | These tools already format flows correctly; raw bpf parsing is signal complexity for zero value. |
| Outbound traffic blocking | iptables-style rules from scratch | `pfctl -a` anchors with documented file fragments | pf has 25+ years of OpenBSD lineage; rolling rules from Apple's BSD docs is fine, hand-rolling a kernel filter is not. |
| Threat model formalisation | DIY taxonomy | STRIDE (Microsoft) primary + LINDDUN (KU Leuven) overlay | STRIDE is industry standard; LINDDUN was specifically designed to fill STRIDE's privacy blind spot. |
| Framework gap mapping | Free-form prose | NIST 800-53 control catalogue + per-control "Met / Partial / Not Pursued / N/A" structure | Auditors expect this structure; deviation reads as evasion. |
| Cryptographic verification of model file | Custom hash | Existing setup.sh `MODEL_SHA256` constant (Step 5) + `shasum -a 256` | Already proven pattern in this codebase. |

**Key insight:** The Phase 2.7 work is in **assembling well-known tools into an auditable narrative**, not in building any new tooling. The auditor credibility comes from using off-the-shelf federally-recognised tooling (Syft, pf, NIST 800-53) — anything hand-rolled would be flagged as not-third-party-validated and the entire posture claim degrades.

## State of the Art per Research Priority

### Priority 1: Threat Model Methodology

**Decision: STRIDE primary + LINDDUN privacy overlay (HIGH confidence).**

For a privacy-first tool with gov / healthcare / journalist audience, STRIDE alone leaves a measurable gap. STRIDE was designed for security-property threats (Spoofing, Tampering, Repudiation, Information Disclosure, DoS, Elevation of Privilege) — it does NOT cover **privacy-by-design** failures where the system functions exactly as designed but the design itself harms privacy (e.g., clipboard managers retaining transcripts; vocab.txt leaking client names). LINDDUN was developed at KU Leuven specifically to address this: Linkability, Identifiability, Non-repudiation, Detectability, Disclosure of information, Unawareness, Non-compliance.

**Research finding (HIGH confidence):** LINDDUN was designed to be **layered on top of STRIDE**, not as a replacement. "Teams already familiar with security threat modeling approaches such as STRIDE will find it easy to adopt LINDDUN, as it follows the same principles. As privacy and security threat modeling have much in common and require the same system model as a starting point, it is possible to perform both in parallel" (linddun.org / Security Compass blog).

### Priority 2: pf + tcpdump on macOS Sequoia 15.7.5

**STATUS: HIGH confidence on pf rule mechanics, MEDIUM-LOW confidence that pf rules apply consistently on macOS 15.x. CRITICAL Sequoia regression documented (Pitfall 1 below).**

**The mechanics (HIGH confidence):**

macOS pf is a port of OpenBSD pf. Rules are managed via `pfctl`. Anchors (named sub-rulesets) are the primary scoping mechanism. **pf cannot scope rules by PID directly** — it scopes by user (UID), group (GID), interface, or socket attributes. The standard pattern for "block this process's network access" is:

1. Run the target process as a dedicated UID (or check current UID via `id -u`).
2. Create a pf anchor with rules scoped to that UID.
3. Load the anchor.
4. Capture on `pflog0` (pf's logging interface) plus all active interfaces.

**Recommended pf rule fragment for `tests/security/verify_egress.sh`:**

```bash
# /tmp/purplevoice-egress.pf — generated by verify_egress.sh, loaded into anchor
# Block all outbound TCP/UDP for the current UID + log to pflog0.
PURPLEVOICE_UID=$(id -u)
cat > /tmp/purplevoice-egress.pf <<EOF
block log out quick proto { tcp udp } from any to any user $PURPLEVOICE_UID
EOF

# Load into a named anchor (so we don't disturb existing pf state)
sudo pfctl -a purplevoice-egress -f /tmp/purplevoice-egress.pf
sudo pfctl -e  # enable pf if not already
```

**Capture pattern:**

```bash
# Run tcpdump in background on pflog0 (pf-blocked traffic) AND on the active
# interface (en0 for Wi-Fi, en1 for Ethernet) for 10 seconds during the
# recording window.
sudo tcpdump -nn -i pflog0 -w /tmp/purplevoice-egress-pflog.pcap &
PFLOG_PID=$!
sudo tcpdump -nn -i any -w /tmp/purplevoice-egress-any.pcap \
  "(tcp or udp) and not (host 127.0.0.1 or host ::1)" &
ANY_PID=$!
sleep 0.5  # let tcpdump initialise

# Synthesise a recording (sources lib/process_tree.sh)
synthesise_recording 2.0
RECORD_PID=$!
wait "$RECORD_PID"

# Stop tcpdumps
sudo kill "$PFLOG_PID" "$ANY_PID" 2>/dev/null
wait "$PFLOG_PID" "$ANY_PID" 2>/dev/null

# Parse: filter by purplevoice process tree PIDs
# (NB: tcpdump doesn't natively filter by PID — the parse step joins
#  pcap output with a parallel lsof snapshot taken during the capture window.)
```

**The Sequoia regression (MEDIUM confidence on the *severity*; HIGH confidence the regression EXISTS):**

Apple Developer Forums thread 776914 documents that on macOS 15.x (15.0.0 - 15.3.1), **outgoing unicast packets bypass pf rules** despite `block all` configuration. Confirmed: even `sudo pfctl -e -F all -f ./pf.conf` with `block all` allows packets through from `com.apple.geod`, Safari, etc. Apple DTS engineer acknowledged the issue but did not provide a workaround. Unclear from public sources whether 15.7.5 (Oliver's machine) still has this regression or whether it's been fixed in a 15.x point release. The mjtsai.com blog (2024-09-18) catalogues the broader Sequoia firewall regressions — including DNS response blocking issues, GUI disconnection from rules, and silent settings reversion on upgrade.

**Implication for D-05:** The pf-based primary egress proof may be unreliable on the exact OS the user runs. **The planner MUST design `verify_egress.sh` to work even when pf is silently broken** — the 3-layer evidence chain (lsof → nettop → pf+tcpdump) is the answer. If pf fails, the lsof + nettop layers still prove process-tree silence (just from socket-state evidence rather than packet-flow evidence). The README + SECURITY.md must acknowledge this pf caveat with a documented test of pf efficacy itself (a positive control: try to `curl example.com` as the test UID under the anchor; if the curl succeeds, pf is broken and the fallback is the load-bearing evidence).

**Sudo handling (HIGH confidence):**

```bash
# At top of verify_egress.sh
if ! sudo -n true 2>/dev/null; then
  echo "verify_egress.sh: sudo required for pf + tcpdump." >&2
  echo "  Run interactively (not under CI) or pre-authenticate sudo." >&2
  echo "  Layers 1-2 (lsof + nettop) will still run; layer 3 (pf+tcpdump) skipped." >&2
  PF_AVAILABLE=0
else
  PF_AVAILABLE=1
fi
```

The script proceeds with layers 1-2 unconditionally; layer 3 is conditional on sudo. Exit non-zero only if any layer surfaces a leak. The README documents sudo as required for the strongest claim; CI / non-interactive contexts get the weaker (but still meaningful) lsof + nettop claim.

### Known limitation: Homebrew is not air-gap-friendly

Homebrew's typical install path requires network access for cask + bottle download. For air-gapped deployment, two options:

1. **Sideload Homebrew bottles**: on a connected reference machine, `brew fetch sox whisper-cpp --bottle` produces tarballs at `~/Library/Caches/Homebrew/downloads/`. USB-transfer these to the target and `brew install <local-bottle>.tar.gz`.

## Pitfalls

### Pitfall 1: macOS Sequoia 15.7.5 pf regression silently breaks the primary egress proof

**What goes wrong:** Apple Developer Forums thread 776914 documents that on macOS 15.x (verified affected: 15.0.0-15.3.1; status on 15.7.5 unconfirmed by public sources), pf rules with `block all` are silently bypassed by some processes. `pfctl -e -F all -f ./pf.conf` returns success; `tcpdump` shows packets going out from `com.apple.geod`, Safari, etc. The egress proof (`verify_egress.sh`) returns "PASS" because **no purplevoice-attributable packets were captured**, but the planner / auditor cannot distinguish between "PurpleVoice silent because PurpleVoice doesn't egress" and "PurpleVoice silent because pf is broken so we have no enforcement at all".

**Why it happens:** Apple-maintained fork of OpenBSD pf has had multiple Sequoia regressions per mjtsai.com 2024-09-18 catalogue (DNS response blocking, GUI disconnection from rules, silent settings reversion on upgrade). The Apple DTS engineer in thread 776914 acknowledged the issue but provided no documented workaround.

**How to avoid:**
1. **Build a positive control into `verify_egress.sh`** — before declaring PASS, the script attempts a `curl --max-time 2 https://example.com` from the same UID under the pf anchor. If the curl SUCCEEDS, the test FAILS with "pf is not enforcing — fall back to lsof/nettop layers; egress claim weakened."
2. **Use the 3-layer evidence chain** — lsof + nettop + (pf + tcpdump). If pf is broken, layers 1-2 still prove process-tree silence (just from socket-state evidence). The README + SECURITY.md acknowledge this caveat with "the pf layer adds packet-flow evidence on macOS versions where pf is functional; lsof + nettop provide socket-state evidence on all versions."
3. **Document the caveat explicitly in SECURITY.md** — auditors must know the test is doing something subtle on Sequoia.

**Warning signs:** `verify_egress.sh` always passes even after deliberately introducing a `curl example.com` into purplevoice-record (which should fail the test). If the test passes when it shouldn't, pf isn't enforcing — fall back to the lsof/nettop layers as primary.

### Pitfall 2: Syft scan against `~/` leaks unrelated packages into SBOM

**What goes wrong:** Naive `syft scan dir:$HOME` produces an SBOM listing every Homebrew formula installed on Oliver's machine plus every .git'd project in his home directory. This is both privacy-leaky and audit-noisy — auditors would reasonably ask "why is this random Rails app a dependency of PurpleVoice?"

**Why it happens:** Syft scans recursively and detects every package ecosystem it can. Without explicit scope, "PurpleVoice's TCB" and "everything on Oliver's laptop" become the same SBOM.

**How to avoid:**
- Scope Syft scans explicitly. Recommended: `syft scan dir:$REPO_ROOT` (the repo only) + a separate `syft scan dir:/opt/homebrew/Cellar/{sox,whisper-cpp}` (named-formula scope only) + a separate `syft scan dir:/Applications/Hammerspoon.app`.
- Merge results post-hoc OR run multiple Syft invocations and combine via `syft convert`.
- The planner picks a single primary scan invocation for setup.sh Step 8; the multi-scope invocation is documented as a release-only "comprehensive SBOM regen" target for thoroughness.

### Pitfall 3: SBOM.spdx.json drift from setup.sh regen creates noisy git diffs

**What goes wrong:** Syft output includes timestamps + scan-IDs that change every run. `setup.sh` Step 8 regenerates `SBOM.spdx.json` with a fresh `creationInfo.created` timestamp and a fresh `documentNamespace` UUID — every run produces a diff even if no packages changed. `git status` is permanently dirty after every setup.sh run. The user becomes blind to *meaningful* SBOM drift (e.g., a brew upgrade that changed sox's version) because *every* regen produces noise.

**Why it happens:** SPDX 2.3 spec requires creationInfo.created (ISO 8601 timestamp) and documentNamespace (URI, conventionally with a UUID).

**How to avoid:**
- **Option A (recommended):** post-process SBOM.spdx.json to **deterministicise** the volatile fields. After Syft writes the file, jq-rewrite `creationInfo.created` to a constant ("derived from package set, not regen time") and `documentNamespace` to a deterministic URI based on `git rev-parse HEAD`. Now the file only changes when the package set changes.
- **Option B:** commit SBOM.spdx.json once per release, regen only at release-time (manual step), not on every setup.sh run. **Conflicts with D-12** which says regen on every setup.sh — but the conflict is reconcilable: D-12 says "regenerates if Syft present" not "writes a new file every time"; if the deterministicised content is identical, no diff appears and idempotency is preserved.
- **Option C:** add SBOM.spdx.json to a `.gitattributes` filter to strip volatile fields on commit. Most complex; reject in favour of A.

**Recommendation:** Option A. The post-process step is ~10 lines of jq.

### Pitfall 4: "compatible with" → "compliant" drift across edits

**What goes wrong:** Author writes "compatible with HIPAA Security Rule" in v1; a later iteration condenses to "HIPAA-compatible" then "HIPAA-compliant"; by v3 SECURITY.md is making over-claims. D-17 mandates consistency.

**How to avoid:**
- `tests/test_security_md_framing.sh` lint: `grep -nE '\b(compliant|certified|guarantees)\b' SECURITY.md | grep -vE '(not compliant|not certified|does not guarantee|"compliant"|"certified")'` — any uncovered hit fails the build.
- Plan-checker greps for the same patterns at every revision.
- The lint runs in the FUNCTIONAL suite (`tests/run_all.sh`), not the security suite, so it gates per-commit.

### Pitfall 5: Egress test runs while Hammerspoon menubar process holds a TCP keepalive connection unrelated to PurpleVoice

**What goes wrong:** Hammerspoon may hold long-lived TCP connections (Sparkle update check, websocket to a dev tool, etc.). If the egress test scopes by UID, those connections appear in the lsof / nettop snapshots and fail the test even though they're not PurpleVoice-attributable.

**How to avoid:**
- D-06 explicitly scopes the assertion to **the PurpleVoice process tree** (purplevoice-record + sox + whisper-cli), NOT Hammerspoon's main process. The lsof / nettop / tcpdump filters use `-p <pid>` lists derived from `purplevoice_pid_tree()` — Hammerspoon's main PID is intentionally excluded.
- The system-wide pflog0 capture (D-06) is included for transparency but the assertion is on the process-tree subset.

### Pitfall 6: synthesise_recording() introduces a second whisper-cli invocation site, breaking Pattern 2

**What goes wrong:** `synthesise_recording()` in `tests/security/lib/process_tree.sh` needs to drive `purplevoice-record` end-to-end. If the helper adds its own `whisper-cli` call (e.g., to validate the synth WAV before feeding), it violates `grep -c WHISPER_BIN purplevoice-record == 2` — the Pattern 2 boundary that v1.1 warm-process upgrade depends on.

**How to avoid:**
- `synthesise_recording()` does NOT call whisper-cli. It pre-stages a WAV using `silence_wav` from the existing `tests/lib/sample_audio.sh` and invokes `purplevoice-record` with `PURPLEVOICE_TEST_SKIP_SOX=1`. The bash glue's existing `transcribe()` then calls whisper-cli — that call is the SOLE invocation site, preserved.

### Pitfall 7: setup.sh Step 8 SBOM regen on every run breaks the no-spurious-regen invariant if Syft scans are slow

**What goes wrong:** Syft scans of `/opt/homebrew/Cellar` can take 5-30 seconds. setup.sh's idempotency guarantee is "safe to re-run" — but if every re-run takes 30s of SBOM regen, the user stops re-running setup.sh in practice.

**How to avoid:**
- Step 8 SBOM regen runs only if **a precondition file (e.g., setup.sh content hash, brew Cellar mtime) has changed since the last regen**. Cache the hash in `~/.cache/purplevoice/sbom-source-hash.txt`; skip regen if unchanged.
- Alternatively: D-12 says "if Syft present" not "always" — interpret D-12 as "Syft present + drift signal present". Drift signal: any of (Cellar mtime newer than SBOM.spdx.json mtime, repo HEAD different from cached, force flag set).
- Reasonable scope for this phase: skip the optimisation; let the scan run on every setup.sh; document the 5-30s latency in setup.sh banner. The optimisation is a low-priority follow-up.

### Pitfall 8: Hammerspoon's brew cask install in OFFLINE mode is structurally impossible without sideload

**What goes wrong:** `PURPLEVOICE_OFFLINE=1` is supposed to make setup.sh fully offline-capable. But `brew install --cask hammerspoon` requires network. There's no graceful path that keeps Hammerspoon installation automatic in OFFLINE mode.

**How to avoid:**
- Acknowledge this honestly in SECURITY.md and the README. The OFFLINE mode handles the WHISPER MODEL + SILERO VAD downloads (the parts most operators want air-gapped); Hammerspoon installation in air-gapped environments requires manual `.dmg` copy from a connected machine.
- Document the manual install: download Hammerspoon-1.1.1.zip from www.hammerspoon.org on connected machine, USB-transfer, drag to /Applications, then run `PURPLEVOICE_OFFLINE=1 setup.sh` which verifies presence and skips the brew step.

### Pitfall 9: Naming collision — `tests/run_all.sh` vs `tests/security/run_all.sh`

**What goes wrong:** Two files named `run_all.sh`. The Phase 3 README, the release-gate runbook, and external auditor instructions all need to disambiguate. Easy to write "run `tests/run_all.sh`" when meaning the security suite.

**How to avoid:**
- README's "verify the security claims" section uses **explicit full paths**, never bare "run_all.sh".
- The aggregator log files differentiate: `/tmp/purplevoice-test.log` (functional) vs `/tmp/purplevoice-security-test.log` (security).
- The aggregator print prefixes differentiate: `[test]` (functional) vs `[security]` (security).

### Pitfall 10: "Audit walks the gap analysis backwards" — different audiences, different entry points

**What goes wrong:** SECURITY.md's structure assumes a linear reader. A US federal IT auditor reads NIST 800-53 first; an EU institutional buyer reads ISO 27001 first; a journalist reads the threat model first. If sections are deeply nested, each audience hits friction.

**How to avoid:**
- SECURITY.md opens with a **1-page TL;DR** for non-technical readers (per D-13's "this audience covers other named audiences").
- Followed by a **table-of-contents with audience-specific entry-points**: "If you're a journalist, start here. If you're a federal IT auditor, jump to NIST 800-53. If you're an EU institutional buyer, jump to ISO 27001. If you're a procurement officer, jump to the SBOM."
- Each gap-analysis section is **self-contained** — readable without first reading the threat model.
- The hybrid prose+table format (D-15) supports skim and depth simultaneously.

### Pitfall 11: SOC 2 / ISO 27001 framing accidentally implies organisational certification

**What goes wrong:** A sentence like "PurpleVoice is compatible with SOC 2 Type II for the in-scope code" is technically careful but parses to a casual reader as "PurpleVoice has SOC 2." Reuse of words ("Type II", "Annex A") cues compliance; over-time reading makes the disclaimer fade.

**How to avoid:**
- Open every framework section with the explicit disclaimer **first**: "SOC 2 Type II reports are issued to organisations after a CPA-led audit. PurpleVoice is a tool, not an organisation, and is not SOC 2 audited. The framing below describes how PurpleVoice's design supports an organisation's SOC 2 posture." — same shape as Tailscale's "Tailscale can be a supporting safeguard for your HIPAA-compliant system" framing.
- Repeat the qualifier at every list of controls: "**(supporting safeguard, not certification)**" parenthetical after the framework name.

### Pitfall 12: Air-gap test requires deleting & restoring a 488 MB model file

**What goes wrong:** `verify_air_gap.sh` Invariant 2 requires `mv` of the Whisper model out and back to test the missing-file path. On heavy use, this churn could corrupt the file (interrupted at the wrong moment) or leak the model to /tmp where it doesn't belong.

**How to avoid:**
- Use `mv` (atomic on same filesystem) not `cp + rm`. The `/tmp/saved-model` move is single-syscall.
- Add a trap: `trap 'mv /tmp/saved-model "$MODEL" 2>/dev/null || true' EXIT` — guarantees restoration even if test crashes mid-run.
- Document the test's required disk space (488 MB temporary in /tmp) in the script's preamble.

### Pitfall 13: NIST 800-53 control IDs drift between Rev 4 and Rev 5

**What goes wrong:** Older auditor checklists reference Rev 4 IDs (e.g., AC-3 vs AC-3a-c subdivision). Mixing Rev 4 and Rev 5 IDs across SECURITY.md sections produces silent errors.

**How to avoid:**
- D-14 explicitly locks Rev 5. SECURITY.md's NIST section opens with "All control IDs reference NIST SP 800-53 Rev 5 (csrc.nist.gov/pubs/sp/800/53/r5/upd1/final)."
- The plan-checker does a sanity grep for Rev-4-only enhancements (`AC-3(7)`, `SC-12(2)` etc. that don't exist in Rev 5) and flags.

### Pitfall 14: Reproducible build (SEC-05) over-promises

**What goes wrong:** Section claims "build is reproducible" without caveat; reader runs `setup.sh`, gets a different binary SHA256 (because Xcode CLT version differs), and concludes the project is dishonest.

**How to avoid:**
- Section opens with: "Full byte-identical reproducibility for whisper.cpp Metal-compiled artefacts is **not currently achievable** — Whisper.cpp's Metal compilation is sensitive to Xcode CLT version, macOS SDK version, and even compiler revision. The reproducibility we DO have: (1) source code is git-tracked; (2) brew bottles are SHA256-verified by Homebrew; (3) the GGML model file SHA256 is pinned in setup.sh and verified on every install; (4) the `voice-cc-record` + `purplevoice-lua/init.lua` files are plain text and reviewable. Best-effort byte-level reproducibility (re-run setup.sh on a clean machine, compare SHAs) is documented; toolchain-version drift is acknowledged."
- This is the framing CONTEXT.md's Deferred Ideas section already endorses.

### Pitfall 15: Plan-checker doesn't catch missing "compatible with" qualifier in tables (only prose)

**What goes wrong:** Lint regex catches "compliant" in prose but the gap-analysis table's "Status" column has cells like "Compliant" without context. Tables get less scrutiny than prose during review.

**How to avoid:**
- Table values use the standardised vocabulary: **Met / Partial / Not Pursued / N/A**, never "Compliant" or "Certified".
- Lint regex covers tables: `awk` over `SECURITY.md` looking for `|.*\b(compliant|certified)\b.*|` patterns inside markdown table rows.

### Example 6: SPDX 2.3 annotation injection for system context

```bash
# Pattern: post-process Syft SPDX-JSON to inject 4 system-context annotations
# Source: spdx.github.io/spdx-spec/v2.3/annotations + jq cookbook
# (HIGH confidence — annotations are SPDX-2.3-spec-defined, federally recognised)

inject_system_context() {
  local sbom="$1"
  local timestamp; timestamp=$(date -u +"%Y-%m-%dT%H:%M:%SZ")
  local macos_version; macos_version=$(sw_vers -productVersion)
  local macos_build; macos_build=$(sw_vers -buildVersion)
  local arch; arch=$(uname -m)
  local clt_version; clt_version=$(pkgutil --pkg-info=com.apple.pkg.CLTools_Executables 2>/dev/null | awk -F': ' '/^version:/ {print $2}')
  local brew_version; brew_version=$(brew --version | head -1 | awk '{print $2}')

  # Inject 5 annotations into SBOM.spdx.json's top-level annotations array
  jq --arg ts "$timestamp" \
     --arg mv "macOS-version=$macos_version ($macos_build)" \
     --arg arch "hardware-platform=$arch (Apple Silicon)" \
     --arg clt "xcode-clt-version=$clt_version" \
     --arg brew "brew-version=$brew_version" \
     '.annotations += [
        {annotationDate: $ts, annotationType: "OTHER", annotator: "Tool: PurpleVoice-setup.sh", comment: ("system-context: " + $mv)},
        {annotationDate: $ts, annotationType: "OTHER", annotator: "Tool: PurpleVoice-setup.sh", comment: ("system-context: " + $arch)},
        {annotationDate: $ts, annotationType: "OTHER", annotator: "Tool: PurpleVoice-setup.sh", comment: ("system-context: " + $clt)},
        {annotationDate: $ts, annotationType: "OTHER", annotator: "Tool: PurpleVoice-setup.sh", comment: ("system-context: " + $brew)}
      ]' \
     "$sbom" > "$sbom.tmp" && mv "$sbom.tmp" "$sbom"
}
```

|--------------|------------------------------|--------|
